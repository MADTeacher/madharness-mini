# Учебная карта madharness-mini

`madharness-mini` — сквозной проект для курса и книги по harness-инженерии.
Код развивается по веткам: каждая ветка фиксирует одну главу и даёт студенту
состояние харнесса, которое можно читать, запускать, ломать и улучшать.

## Как пользоваться маршрутом

1. Переключитесь на нужную ветку.
2. Прочитайте `README.md` этой ветки.
3. Откройте `LABS.md` и выберите задачу нужного уровня.
4. Используйте `docs/README.md` как локальный индекс по документации ветки.
5. После правки запускайте тесты и смотрите трассу.

```bash
git switch 03-Context-Layer
uv run -m unittest discover -s tests
```

## Ветки и главы

| Ветка | Глава | Что появляется | Лабораторный фокус |
| --- | --- | --- | --- |
| `01-minimalistic-harness` | Минимальный харнесс | CLI, обращения к модели, инструменты, политика, трасса | Понять цикл: модель — инструмент — observation. |
| `02-AGENTS-md` | Проектные инструкции | `AGENTS.md`, nested rules, `read_image` | Управлять локальными правилами проекта. |
| `03-Context-Layer` | Слой контекста | `ContextManager`, fragments, budget, `context_report` | Видеть и ограничивать запрос к модели. |
| `04-Agents-Skills` | Agent Skills | `SKILL.md`, каталог, `activate_skill`, ресурсы | Добавлять рабочие инструкции без кода инструментов. |
| `05-mcp` | MCP-инструменты | `mcp.json`, stdio-клиент, `tools/list`, `tools/call` | Подключать внешние инструменты как инструменты модели. |
| `06-subagents` | Субагенты | роли, `delegate_task`, `ask_user`, дочерние трассы | Делегировать задачи с ограниченными полномочиями. |
| `07-hooks` | Hooks | события жизненного цикла, command hooks, redaction, blocking | Добавлять аудит и проектную политику поверх харнесса. |

## Формат лабораторных

В каждой ветке `LABS.md` делится на три уровня:

- базовый уровень: маленькие задачи для чтения кода и точечной правки;
- основной уровень: задачи, которые меняют поведение текущего механизма;
- продвинутый уровень: задачи с архитектурным выбором и заметным компромиссом.

В каждом уровне должно быть не меньше двух задач, обычно три. Оценки времени
не используются: одна и та же лабораторная может быть быстрой для опытного
разработчика и глубокой для студента, который впервые видит харнесс.

Каждая задача описывает:

- цель;
- задание;
- ограничения;
- критерий готовности;
- проверку или вопросы после выполнения.

## Рекомендуемый ритм занятия

1. Коротко разобрать `README.md` ветки.
2. Ознакомиться с основным потоком работы харнесса в `docs/code-overview.md`.
3. Выбрать одну базовую задачу всем студентам.
4. Дать основной или продвинутый уровень как самостоятельную работу.
5. В конце сравнить трассы: какие обращения к модели, вызовы инструментов и
   ограничения были реально задействованы.

## Проверка проекта

Во всех ветках базовая команда проверки одна:

```bash
uv run -m unittest discover -s tests
```

Если конкретная лабораторная требует ручной проверки CLI, укажите это в
своём `LABS.md`.
