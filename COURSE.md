# Учебная карта madharness-mini

`madharness-mini` — сквозной проект для курса и книги по harness-инженерии.
Код развивается по веткам: каждая ветка фиксирует одну главу и даёт студенту
состояние harness, которое можно читать, запускать, ломать и улучшать.

## Как пользоваться маршрутом

1. Переключитесь на нужную ветку.
2. Прочитайте `README.md` этой ветки.
3. Откройте `LABS.md` и выберите задачу нужного уровня.
4. Используйте `docs/README.md` как локальный индекс по документации ветки.
5. После правки запускайте тесты и смотрите trace.

```bash
git switch 03-Context-Layer
uv run -m unittest discover -s tests
```

## Ветки и главы

| Ветка | Глава | Что появляется | Лабораторный фокус |
| --- | --- | --- | --- |
| `01-minimalistic-harness` | Минимальный harness | CLI, model call, tools, policy, trace | Понять цикл model/tool/observation. |
| `02-AGENTS-md` | Проектные инструкции | `AGENTS.md`, nested rules, `read_image` | Управлять локальными правилами проекта. |
| `03-Context-Layer` | Слой контекста | `ContextManager`, fragments, budget, `context_report` | Видеть и ограничивать запрос к модели. |
| `04-Agents-Skills` | Agent Skills | `SKILL.md`, catalog, `activate_skill`, resources | Добавлять workflow-инструкции без кода tools. |
| `05-mcp` | MCP tools | `mcp.json`, stdio client, `tools/list`, `tools/call` | Подключать внешние инструменты как обычные tools. |
| `06-subagents` | Субагенты | роли, `delegate_task`, `ask_user`, child traces | Делегировать задачи с ограниченными полномочиями. |
| `07-hooks` | Hooks | lifecycle events, command hooks, redaction, blocking | Добавлять аудит и проектную политику поверх harness. |

## Формат лабораторных

В каждой ветке `LABS.md` делится на три уровня:

- базовый уровень: маленькие задачи для чтения кода и точечной правки;
- основной уровень: задачи, которые меняют поведение текущего механизма;
- продвинутый уровень: задачи с архитектурным выбором и заметным trade-off.

В каждом уровне должно быть не меньше двух задач, обычно три. Оценки времени
не используются: одна и та же лабораторная может быть быстрой для опытного
разработчика и глубокой для студента, который впервые видит harness.

Каждая задача описывает:

- цель;
- задание;
- ограничения;
- критерий готовности;
- проверку или вопросы после выполнения.

## Методический принцип

Документация ветки должна описывать только то, что реально есть в коде этой
ветки. Будущие механизмы можно упоминать только как краткий переход в следующую
главу, но не как доступную возможность.

Это правило защищает студента от ложной навигации: если в `03-Context-Layer`
написано про MCP как текущую функцию, студент будет искать несуществующий код.
Поэтому каждый `docs/README.md` является локальным индексом именно своей ветки.

## Рекомендуемый ритм занятия

1. Коротко разобрать `README.md` ветки.
2. Показать основной поток выполнения в `docs/code-overview.md`.
3. Выбрать одну базовую задачу всем студентам.
4. Дать основной или продвинутый уровень как самостоятельную работу.
5. В конце сравнить trace: какие model calls, tool calls и ограничения были
   реально задействованы.

## Проверка проекта

Во всех ветках базовая команда проверки одна:

```bash
uv run -m unittest discover -s tests
```

Если конкретная лабораторная требует ручной проверки CLI, она указывает это в
своём `LABS.md`.
