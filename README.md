# madharness-mini

`madharness-mini` — учебный минималистичный харнесс для работы кодирующего ИИ-агента с локальным программным продуктом. Он даёт модели понятный цикл разработки: прочитать проект, найти нужный код, внести точечные изменения, запустить проверку и записать трассу того, что произошло.

Проект написан для Python 3.13+ и не имеет runtime-зависимостей. Внутри используется OpenAI-совместимый API `/chat/completions`, поэтому можно подключить OpenRouter, KodikRouter, локальный совместимый сервер или другой сервис с тем же форматом API.

## Когда использовать

`madharness-mini` полезен, когда вы хотите дать агенту контролируемый доступ к локальному проекту:

- разобраться в существующей кодовой базе;
- добавить небольшую функцию;
- исправить баг;
- написать или обновить тесты;
- провести ревью изменений;
- обновить документацию;
- выполнить учебный разбор того, как устроен агентский цикл.

Харнесс не заменяет IDE, Git и CI. Он работает как маленький слой между моделью и рабочей папкой: модель видит задачу, инструкции проекта и набор инструментов, а харнесс проверяет файловые пути, shell-команды, контекст и журнал выполнения.

## Установка

Установите команду из GitHub:

```bash
uv tool install madharness-mini --from git+https://github.com/MADTeacher/madharness-mini.git
```

Если терминал после установки не видит `madharness-mini`, обновите shell path и откройте терминал заново:

```bash
uv tool update-shell
```

## Подключение проекта

Перейдите в корень продукта, с которым должен работать агент:

```bash
cd /path/to/your/product
```

Создайте локальную настройку с настройками по умолчанию:

```bash
madharness-mini init
```

По умолчанию харнесс использует OpenRouter и модель `deepseek/deepseek-v4-flash`. Если команда запущена в интерактивном терминале и ключ ещё не задан, она попросит API-ключ. Ключ можно сразу не вводить и позже записать в `.madharness-mini/config.json` через повторный `init --api-key ...` или ручную правку файла.

Если нужно сразу указать другой API, модель или ключ, передайте их явно:

```bash
madharness-mini init \
  --base-url https://openrouter.ai/api/v1 \
  --model deepseek/deepseek-v4-flash \
  --api-key "ключ-доступа-openrouter"
```

Команда создаёт `.madharness-mini/config.json` и каталог `.madharness-mini/traces`. Эти файлы относятся к конкретному продукту, а не к глобальной установке харнесса. Для обычного использования храните модель, адрес API и ключ именно в `.madharness-mini/config.json`.

## Первые команды

Задать вопрос без доступа к инструментам:

```bash
madharness-mini ask "Объясни, что делает этот проект"
```

Запустить агентский режим с чтением файлов, поиском, правками и проверками:

```bash
madharness-mini run "Найди команду для запуска тестов и объясни, что она проверяет"
```

Попросить агента выполнить реальную задачу:

```bash
madharness-mini run "Добавь в CLI флаг --version. Сначала найди точку входа, затем измени код и запусти подходящие тесты."
```

Посмотреть краткую сводку трассы после запуска:

```bash
madharness-mini trace 20260529-171000
```

Идентификатор трассы можно брать из строки `Trace: ...`, которую `ask` и `run` печатают после ответа.

## Как применять к разработке продуктов

Один и тот же харнесс можно использовать для разных типов программных продуктов: Python-пакета, CLI, веб-приложения, backend-сервиса, мобильного клиента, учебного проекта или внутренней утилиты. Важно не название стека, а наличие понятной рабочей папки, правил проекта и проверочных команд.

Рекомендуемый цикл:

1. Запускайте `madharness-mini` из корня продукта.
2. Опишите агенту конкретную задачу и ожидаемую проверку.
3. Попросите сначала изучить существующие файлы и стиль проекта.
4. Разбивайте крупную работу на небольшие запуски.
5. После правки просите выполнить тест, build, lint или smoke-check, если такая команда есть в проекте.
6. Смотрите итоговый ответ и trace, если нужно понять ход рассуждений и вызовы инструментов.

Примеры задач для разных продуктов:

```bash
madharness-mini run "В Python-библиотеке добавь поддержку нового параметра timeout. Найди существующие тесты и обнови их."
```

```bash
madharness-mini run "В frontend-приложении исправь состояние загрузки на странице профиля. После правки запусти npm test."
```

```bash
madharness-mini run "В Go-сервисе найди обработчик /health, добавь поле version в JSON-ответ и запусти go test ./..."
```

```bash
madharness-mini run "В Flutter-приложении обнови текст пустого состояния и запусти flutter test."
```

```bash
madharness-mini run "Обнови README так, чтобы новый разработчик понял установку, запуск, тесты и типичный workflow."
```

Хорошая формулировка задачи обычно содержит:

- что нужно изменить;
- где искать, если вы знаете область;
- какие ограничения соблюдать;
- какую команду проверки запустить;
- что делать, если проверка недоступна.

Например:

```bash
madharness-mini run "Исправь падение при пустом config.json. Не добавляй зависимости. После изменения запусти uv run -m unittest discover -s tests. Если тесты не стартуют из-за окружения, объясни причину."
```

Если проверочная команда требует пайпы, редиректы или цепочки `&&`, вынесите её в скрипт проекта и просите агента запускать одну простую команду. Инструмент `run_shell` намеренно принимает только одну безопасную команду с аргументами.

## Команды CLI

| Команда | Что делает | Когда использовать |
| --- | --- | --- |
| `madharness-mini init` | Создаёт или обновляет `.madharness-mini/config.json`. | При первом подключении продукта или смене модели/API. |
| `madharness-mini ask "..."` | Отправляет один вопрос модели без инструментов. | Для объяснений, идей и быстрой проверки доступа к LLM. |
| `madharness-mini run "..."` | Запускает агентский цикл с инструментами workspace. | Для анализа проекта, правок, тестов и ревью. |
| `madharness-mini trace <id>` | Показывает краткую сводку JSONL-трассы. | Чтобы понять, какие шаги были в запуске. |
| `madharness-mini skills list` | Показывает найденные project-local skills. | Чтобы проверить доступные workflow-навыки. |
| `madharness-mini skills show <name>` | Показывает один skill, его metadata и ресурсы. | Для диагностики конкретного навыка. |
| `madharness-mini skills validate` | Печатает ошибки и предупреждения по skills. | Перед использованием новых `SKILL.md`. |
| `madharness-mini subagents list` | Показывает встроенных и project-local субагентов. | Чтобы увидеть роли для делегации. |
| `madharness-mini subagents show <name>` | Показывает настройки и prompt субагента. | Для проверки роли перед запуском. |
| `madharness-mini subagents validate` | Проверяет markdown-файлы субагентов. | После добавления своей роли. |

## Режимы `ask` и `run`

`ask` — один вызов модели. Он подходит для вопросов вроде “объясни архитектуру”, “предложи план” или “что значит эта ошибка”. Инструменты не передаются, поэтому модель не читает файлы и не меняет проект.

`run` — пошаговый агентский цикл. Модель может ответить сразу или вызвать инструмент. Харнесс выполняет инструмент, возвращает результат модели, пишет событие в trace и продолжает цикл до финального ответа или лимита `max_turns`.

Типичный `run` выглядит так:

1. Модель получает задачу и инструкции проекта.
2. Ищет нужные файлы через `list_files` или `search_code`.
3. Читает фрагменты через `read_file`.
4. Вносит точечную правку через `apply_patch`.
5. Запускает безопасную проверку через `run_shell`.
6. Возвращает итог: что изменено, чем проверено, что осталось рискованным.

## Инструменты режима `run`

| Инструмент | Что делает |
| --- | --- |
| `list_files` | Рекурсивно показывает файлы внутри workspace, пропуская `.git`, кэши и служебные каталоги. Возвращает до 200 результатов. |
| `read_file` | Читает UTF-8 фрагмент файла по 1-based строкам и возвращает текст с номерами строк. |
| `read_image` | Проверяет локальное PNG, JPEG, WEBP или неанимированное GIF и прикрепляет изображение к следующему запросу модели, если vision-ввод включён. |
| `search_code` | Ищет буквальную подстроку в файлах проекта. Это не regex и не семантический поиск. |
| `apply_patch` | Применяет точечный патч для создания, изменения, удаления или перемещения файлов. |
| `write_file` | Полностью перезаписывает UTF-8 файл. Для существующих файлов обычно лучше `apply_patch`. |
| `run_shell` | Запускает одну разрешённую команду в workspace или безопасном подкаталоге. Таймаут — 60 секунд. |
| `activate_skill` | Подключает инструкции project-local skill, если skills найдены и пользователь не выбрал skill явно. |
| `delegate_task` | Запускает встроенного или project-local субагента, если режим оркестрации это разрешает. |
| `ask_user` | Доступен только субагентам, которым он явно разрешён; завершает запуск вопросом к пользователю. |

## Проектные инструкции

Для каждого продукта можно добавить файл `AGENTS.md`. Харнесс читает такие файлы перед обращением к модели и добавляет их к системному prompt.

Хороший `AGENTS.md` коротко отвечает на вопросы:

- что это за продукт;
- какой стек используется;
- какие команды запускают тесты, lint и сборку;
- какие директории нельзя менять без причины;
- какие зависимости можно или нельзя добавлять;
- какой стиль кода и комментариев принят;
- какие проверки обязательны перед финальным ответом.

Минимальный пример:

````markdown
# AGENTS.md

## Проект

Это FastAPI-сервис для внутреннего API.

## Правила

- Не добавляй runtime-зависимости без отдельной просьбы.
- Сначала ищи существующие handlers, schemas и tests.
- Для правок backend запускай pytest.
- Не меняй миграции базы данных без явного запроса.

## Команды

```bash
pytest
python -m app
```
````

Если `AGENTS.md` есть в корне и во вложенной папке, сначала применяются правила корня, затем более конкретные вложенные правила. Поддерживается только точное имя `AGENTS.md`; общий объём добавленных проектных инструкций ограничен 32 KiB.

## Конфигурация

Основной файл настроек:

```text
.madharness-mini/config.json
```

Для продукта, с которым работает агент, это главное и рекомендуемое место хранения настроек харнесса, включая `api_key`. Команда `madharness-mini init` создаёт этот файл, а повторный `init` с флагами `--model`, `--base-url` или `--api-key` обновляет соответствующие поля.

Главные поля:

| Поле | Смысл |
| --- | --- |
| `model` | Имя модели, которое отправляется в Chat Completions API. |
| `base_url` | Базовый адрес OpenAI-совместимого API без `/chat/completions` на конце. |
| `api_key` | Ключ доступа к выбранному сервису. |
| `headers` | Необязательные дополнительные HTTP-заголовки. |
| `temperature` | Степень вариативности ответа модели. |
| `max_turns` | Максимальное число ходов модели в `run`. |
| `context_max_tokens` | Примерный бюджет контекста для запроса к модели. |
| `context_keep_recent_turns` | Сколько последних элементов истории стараться сохранять при обрезке контекста. |
| `workspace_root` | Папка, внутри которой агент может читать, писать и запускать команды. Обычно `"."`. |
| `protected_paths` | Пути, к которым файловые инструменты не должны обращаться. |
| `allow_shell` | Включает или выключает `run_shell`. |
| `orchestration_enabled` | Старый общий выключатель делегации; оставлен для совместимости. |
| `orchestration_mode` | Режим оркестрации: `off`, `requested`, `auto`, `required`. |
| `subagent_max_turns` | Лимит ходов субагента, если он не задан в markdown-роли. |
| `subagent_context_max_tokens` | Бюджет контекста субагента, если он не задан в markdown-роли. |
| `supports_image_input` | Разрешает `read_image` прикреплять изображения к запросу модели. |
| `max_image_bytes` | Максимальный размер локального изображения. |
| `image_detail` | Режим детализации изображения: `auto`, `low`, `high`, `original`. |

Технически значения можно переопределить через `.env` или переменные окружения:

| Переменная | Что меняет |
| --- | --- |
| `MADHARNESS_MINI_MODEL` | `model` |
| `MADHARNESS_MINI_BASE_URL` | `base_url` |
| `MADHARNESS_MINI_API_KEY` | `api_key` |
| `MADHARNESS_MINI_ORCHESTRATION_ENABLED` | `orchestration_enabled` |
| `MADHARNESS_MINI_ORCHESTRATION_MODE` | `orchestration_mode` |
| `MADHARNESS_MINI_SUPPORTS_IMAGE_INPUT` | `supports_image_input` |
| `MADHARNESS_MINI_MAX_IMAGE_BYTES` | `max_image_bytes` |
| `MADHARNESS_MINI_IMAGE_DETAIL` | `image_detail` |

Этот слой удобен прежде всего при разработке самого `madharness-mini`, лабораторных работах и временных экспериментах, когда не хочется менять проектный `config.json`. Для обычного подключения продукта держите рабочие настройки в `.madharness-mini/config.json`.

## Адреса LLM API

По умолчанию используется OpenRouter:

```text
https://openrouter.ai/api/v1
```

Примеры базовых адресов:

| Сервис | `base_url` |
| --- | --- |
| OpenRouter | `https://openrouter.ai/api/v1` |
| KodikRouter | `https://kodikrouter.ru/api/v1` |
| Локальный совместимый сервер | `http://127.0.0.1:11434/v1` |
| Свой совместимый API | Любой адрес с OpenAI-совместимым `/chat/completions`. |

Ключ обязателен для запуска `ask` и `run`. Даже если локальный сервер фактически его не проверяет, задайте любое непустое значение в `.madharness-mini/config.json`, например через `madharness-mini init --api-key ...`.

Если провайдер вернул HTTP 429, харнесс один раз учитывает короткий `Retry-After` до 60 секунд. Если повтор тоже не помог, запуск завершается ошибкой с подсказкой проверить лимиты, баланс, модель или ключ.

## Безопасность workspace

Все файловые инструменты проходят через `workspace_root`. Попытка выйти наружу через абсолютный путь или `..` завершается отказом.

По умолчанию защищены:

```json
[".git", ".env", "secrets", "~/.ssh"]
```

`run_shell` дополнительно проверяет команду:

- запрещены управляющие операторы `|`, `>`, `<`, `&&`, `||`, `;`;
- запрещены рискованные фрагменты вроде `rm -rf`, `sudo`, `curl`, `wget`, `ssh`, `scp`, `chmod 777`, `mkfs`, `dd`;
- команда запускается через `subprocess.run` без shell-строки;
- рабочий каталог должен быть внутри workspace;
- таймаут команды — 60 секунд.

Эти ограничения не делают харнесс полноценной песочницей, но задают учебные и практические границы: агент должен работать с локальным продуктом предсказуемо, через маленькие проверяемые действия.

## Трассы

Каждый `ask` и `run` создаёт JSONL-файл:

```text
.madharness-mini/traces/<дата>-<время>-<суффикс>.jsonl
```

В трассу попадают:

- старт и конец сессии;
- вызовы модели;
- отчёт о контекстном бюджете;
- tool calls и observations;
- ошибки инструментов;
- события skills, субагентов, MCP и hooks, если они использовались.

Команда `trace` показывает короткую сводку:

```bash
madharness-mini trace <начало-имени-файла>
```

Trace полезен для отладки: можно увидеть, какие файлы агент читал, почему tool был заблокирован, где закончился бюджет контекста или какая проверка упала.

## Agent Skills

Skills — это project-local инструкции для повторяемых workflow. Они не нужны для обычных задач, но помогают, если в продукте есть специализированный процесс: выпуск релиза, обновление changelog, генерация артефактов, ревью API-контракта.

Харнесс ищет skills в прямых подпапках:

```text
.madharness_mini/skills
.agents/skills
```

Каждый skill — папка с `SKILL.md`. Минимальный пример:

```markdown
---
name: docs-writer
description: Обновляет пользовательскую документацию после изменения поведения.
---

Сначала найди изменённое поведение в коде и тестах.
Затем обнови README или другой явно названный файл.
Не выдумывай команды, если их нет в проекте.
```

Подключить skill явно:

```bash
madharness-mini run "@skill:docs-writer обнови README после изменения CLI"
```

Также поддерживаются маркеры `@skill/name`, `$name` и фразы вида `используй навык name`. Если явного выбора нет, `run` показывает модели компактный каталог skills, и модель может вызвать `activate_skill`.

Ресурсы внутри папки skill, например `scripts/`, `references/` и `assets/`, не читаются заранее. Агент видит их список после активации и может прочитать конкретный файл или запустить documented script, если это разрешено общей политикой.

## Субагенты

Субагенты — это отдельные markdown-роли для делегации внутри `run`. Встроенные роли:

- `researcher` — ищет и объясняет контекст;
- `planner` — составляет план и может менять только markdown-файлы;
- `implementer` — вносит изменения;
- `reviewer` — проверяет риски, регрессии и тесты.

Режим оркестрации задаётся полем `orchestration_mode`:

| Режим | Поведение |
| --- | --- |
| `off` | `delegate_task` не передаётся модели. |
| `requested` | Делегация доступна только если задача явно просит субагентов или оркестрацию. |
| `auto` | Делегация доступна в `run`, модель сама решает, пользоваться ли ей. |
| `required` | Ведущий агент должен координировать работу через субагентов и получает ограниченный набор собственных tools. |

На один запуск режим можно переопределить флагами:

```bash
madharness-mini run --no-orchestrate "Обнови короткий текст"
madharness-mini run --orchestration requested "Используй субагентов для большого рефакторинга"
madharness-mini run --orchestrate-required "Проведи smoke-test researcher/planner/implementer/reviewer"
```

Project-local субагент добавляется файлом:

```text
.madharness-mini/subagents/<name>.md
```

Минимальный формат:

```markdown
---
name: test-writer
description: Пишет минимальные unittest для изменённого кода.
profile: writable
tools: ["list_files", "read_file", "search_code", "apply_patch", "write_file", "run_shell", "ask_user"]
max_turns: 10
---

Ты субагент test-writer.
Пиши маленькие тесты рядом с существующими tests.
```

`profile` описывает характер роли, но реальные полномочия задаёт список `tools`. Субагент не получает `delegate_task`, чтобы не запускать рекурсивную оркестрацию. Если локальная роль использует имя встроенной, она перекрывает встроенную только при `override: true` во frontmatter.

## MCP tools

В `run` можно подключить внешние stdio MCP-серверы. Конфигурация лежит отдельно:

```text
.madharness-mini/mcp.json
```

Минимальный пример:

```json
{
  "servers": {
    "playwright": {
      "enabled": true,
      "command": "npx",
      "args": ["-y", "@playwright/mcp@latest"],
      "cwd": ".",
      "timeout_seconds": 30
    }
  }
}
```

Поддерживаются только stdio transport и MCP tools. Харнесс запускает сервер без shell-строки, вызывает `tools/list`, превращает найденные tools в обычные схемы с именами вида `mcp__playwright__browser_navigate`, а затем отправляет `tools/call` при выборе модели.

`cwd` сервера должен быть внутри workspace. Окружение сервера состоит из безопасного минимума системных переменных и явного `env` из `mcp.json`; ключ LLM API и переменные `MADHARNESS_MINI_*` не передаются автоматически.

## Hooks

Hooks позволяют продукту запускать локальные команды на событиях жизненного цикла. Если `.madharness-mini/hooks.json` отсутствует, hooks выключены.

Минимальный пример:

```json
{
  "hooks": [
    {
      "id": "deny-shell",
      "event": "before_tool_call",
      "match": { "tool": "run_shell" },
      "command": "python3",
      "args": ["scripts/hooks/deny_shell.py"],
      "cwd": ".",
      "timeout_seconds": 3
    }
  ]
}
```

Поддерживаемые события:

| Event | Когда вызывается |
| --- | --- |
| `session_start` | После создания trace и загрузки hooks. |
| `before_model_call` | Перед HTTP-вызовом модели. |
| `after_model_call` | После ответа модели. |
| `before_tool_call` | Перед выполнением инструмента. |
| `after_tool_call` | После результата инструмента. |
| `session_end` | При нормальном завершении. |
| `session_error` | При ошибке сессии. |

Только `before_tool_call` может заблокировать действие. Hook-команда получает JSON события в stdin. Пустой stdout или `{"ok": true}` означает “разрешить”. Ответ `{"ok": false, "block": "причина"}` блокирует tool call, а модель получает обычное observation с `ok=false`.

Команды hooks запускаются списком аргументов, без shell-строки. Их `cwd` должен быть директорией внутри workspace. Секреты не наследуются автоматически; передавайте их явно через поле `env`, если это действительно нужно.

## Работа с изображениями

Инструмент `read_image` доступен всегда, но прикрепляет картинку к запросу модели только если vision-ввод включён в `.madharness-mini/config.json`:

```json
{
  "supports_image_input": true,
  "image_detail": "auto",
  "max_image_bytes": 5000000
}
```

Поддерживаются PNG, JPEG, WEBP и неанимированный GIF. Это удобно для задач вроде “проверь скриншот UI и найди, какой компонент отвечает за проблему”, но итоговая способность зависит от выбранной модели.

## Разработка самого `madharness-mini`

Если вы меняете код этого харнесса, работайте из корня репозитория.

Для лабораторных работ и разработки самого `madharness-mini` можно использовать `.env`, чтобы временно переопределять модель, API-адрес или ключ без изменения `.madharness-mini/config.json`. Для обычного продукта это не основной способ настройки.

Запустить тесты:

```bash
uv run -m unittest discover -s tests
```

Быстро проверить CLI:

```bash
uv run madharness-mini ask "Объясни, что делает этот проект"
```

Переустановить команду из локального checkout:

```bash
uv tool install --python 3.13 --force .
```

Перед изменением поведения держите проект маленьким: не добавляйте runtime-зависимости без явной причины, сохраняйте понятные стандартные структуры Python и проверяйте границы безопасности workspace, shell, traces, skills, MCP и hooks.
