# Дорожная карта Agents Skills

Этот документ описывает, как добавить в `madharness-mini` поддержку агентских навыков без внешних runtime-зависимостей и без plugin discovery. Навыки должны остаться понятным учебным механизмом: markdown-инструкции и, позже, небольшие встроенные provider-объекты, которые подключаются явно.

## Цель

Agents Skill в `madharness-mini` - это именованный набор инструкций и, при необходимости, инструментов, который помогает агенту лучше выполнять узкий класс задач.

Примеры навыков:

| Навык | Что добавляет |
| --- | --- |
| `python-unittest` | Подсказки по запуску и интерпретации `unittest` в проекте. |
| `docs-writer` | Правила для краткой технической документации. |
| `release-notes` | Формат итогов изменений и проверки перед релизом. |

Навык не должен подменять `AGENTS.md`. `AGENTS.md` остаётся правилами конкретного проекта, а skill - повторно используемой способностью харнесса.

## Не цели V1

- Не добавлять внешние зависимости.
- Не исполнять произвольный Python-код из workspace.
- Не делать marketplace, plugin registry или entry points.
- Не подключать навыки автоматически по семантическому поиску.
- Не делать вложенные навыки, зависимости навыков и версионирование пакетов.

## Место в архитектуре

Для V1 достаточно вложенного пакета:

```text
madharness_mini/
  skills/
    __init__.py
    loader.py
    provider.py
    types.py
```

Роли файлов:

| Файл | Роль |
| --- | --- |
| `types.py` | `Skill`, `SkillConfig`, ошибки загрузки. |
| `loader.py` | Читает конфиг, ищет skill directories, валидирует `skill.json` и `SKILL.md`. |
| `provider.py` | Превращает загруженные навыки в `ContextProvider`. |
| `__init__.py` | Экспортирует маленький публичный API пакета. |

В `loop.py` изменение должно быть точечным: передать context providers в `base_context()`.

```python
skills = load_skills(cfg)
context = base_context(cfg, task, providers=[SkillContextProvider(skills)])
```

Сам `ContextManager` уже принимает `ContextProvider`, поэтому слой контекста не нужно переписывать.

## Формат навыка

Навык хранится в обычной папке. В V1 обязательны два файла:

```text
skills/
  docs-writer/
    skill.json
    SKILL.md
```

`skill.json` читается стандартным `json`:

```json
{
  "name": "docs-writer",
  "title": "Documentation Writer",
  "description": "Помогает писать короткую документацию в стиле madharness-mini.",
  "version": 1,
  "entry": "SKILL.md"
}
```

Правила:

- `name` состоит из латинских букв, цифр, `_` и `-`.
- `version` в V1 равен `1`.
- `entry` должен указывать на markdown-файл внутри папки навыка.
- `SKILL.md` читается как UTF-8 с `errors="replace"`.
- Пустой `SKILL.md` считается ошибкой загрузки.
- Один skill-фрагмент ограничивается отдельным лимитом, например 16 KiB.

`SKILL.md` содержит только инструкции для модели:

```markdown
# Documentation Writer

Используй короткие разделы, конкретные пути файлов и проверяемые шаги.
Не превращай документацию в маркетинговый текст.
```

## Конфиг

Предлагаемый формат в `.madharness-mini/config.json`:

```json
{
  "skills": {
    "enabled": ["docs-writer"],
    "paths": ["skills"]
  }
}
```

Поля:

| Поле | Смысл |
| --- | --- |
| `skills.enabled` | Список имён навыков, которые нужно подключить. |
| `skills.paths` | Папки, где искать навыки, относительно `workspace_root`. |

Почему не auto-discovery по всему workspace: навык влияет на системный контекст, поэтому его подключение должно быть явным и проверяемым.

## Загрузка

Алгоритм V1:

1. Взять `skills.paths` из конфига. Если поля нет, навыки выключены.
2. Каждый путь прогнать через `Policy.safe_path()`.
3. Для каждого имени из `skills.enabled` найти папку с таким именем внутри разрешённых путей.
4. Прочитать `skill.json`.
5. Проверить имя, версию, `entry` и выход за папку навыка.
6. Прочитать markdown entry.
7. Вернуть список `Skill`.

Если один навык не загрузился, лучше завершить запуск понятной ошибкой. Молчаливый пропуск опасен: пользователь будет думать, что агент работает с нужной способностью, а модель её не увидит.

## Контекст

`SkillContextProvider` отдаёт по одному `ContextFragment` на навык:

| Поле | Значение |
| --- | --- |
| `id` | `skill:<name>` |
| `source` | путь к `SKILL.md` относительно workspace |
| `placement` | `system` |
| `priority` | после системного промпта и `AGENTS.md`, например `50` |
| `text` | заголовок навыка и markdown-инструкции |

Формат текста:

```markdown
# Skill: docs-writer

<содержимое SKILL.md>
```

Так модель явно видит границу между правилами проекта и подключённой способностью.

## Инструменты навыков

V1 должен быть context-only: навык добавляет только инструкции. Это самый безопасный и понятный шаг.

V2 может разрешить навыкам добавлять инструменты, но не через произвольный код из workspace. Варианты по возрастанию сложности:

| Вариант | Смысл |
| --- | --- |
| Встроенные skill-tools | Инструменты лежат в `madharness_mini.skills.builtin` и подключаются по имени. |
| Declarative wrappers | Навык описывает, какой уже встроенный инструмент использовать с preset-аргументами. |
| Python providers | Только доверенные provider-классы из самого пакета `madharness_mini`, без импорта из workspace. |

Если когда-нибудь понадобится исполнять код навыков из workspace, это должно быть отдельной большой темой безопасности, а не частью первого внедрения.

## CLI

Минимальный CLI можно добавить после loader:

| Команда | Что делает |
| --- | --- |
| `madharness-mini skills list` | Показывает доступные навыки из `skills.paths`. |
| `madharness-mini skills show <name>` | Печатает описание и путь к `SKILL.md`. |

Команду `enable` можно не делать в V1: пользователь может явно редактировать JSON-конфиг, что лучше подходит учебному проекту.

## Безопасность

Границы:

- Skill paths проходят через `Policy.safe_path()`.
- `entry` не может выйти из папки навыка.
- Скрытые и защищённые пути не читаются, если они попадают под `protected_paths`.
- Markdown навыка не исполняется.
- Навык не получает доступ к `api_key`.
- Слишком длинные инструкции обрезаются или дают ошибку до вызова модели.

## Трассы

В трассу стоит добавить событие:

```json
{
  "event": "skills_loaded",
  "skills": ["docs-writer"],
  "count": 1
}
```

Содержимое `SKILL.md` в трассу писать не нужно. Достаточно имён и источников, чтобы не раздувать JSONL.

## Тест-план

Новые тесты:

| Тест | Что проверяет |
| --- | --- |
| `test_skill_loader_reads_enabled_skill` | Валидный skill загружается из `skills.paths`. |
| `test_skill_loader_ignores_when_config_missing` | Без `skills` контекст не меняется. |
| `test_skill_loader_rejects_path_outside_workspace` | Путь навыков не выходит за workspace. |
| `test_skill_loader_rejects_entry_escape` | `entry="../AGENTS.md"` запрещён. |
| `test_skill_context_provider_adds_fragments` | Навык появляется в system-контексте. |
| `test_skill_context_is_limited` | Огромный `SKILL.md` не ломает бюджет контекста. |

## Этапы внедрения

1. Добавить `madharness_mini/skills/types.py` и тесты на структуру `Skill`.
2. Добавить loader с валидацией `skill.json` и `SKILL.md`.
3. Добавить `SkillContextProvider`.
4. Расширить `base_context()` параметром `providers`.
5. Подключить loader в `ask` и `run`.
6. Записать событие `skills_loaded` в трассу.
7. Обновить `docs/capabilities.md` и `docs/code-overview.md`.
8. После стабилизации добавить CLI `skills list/show`.

## Готовность

V1 считается готовой, когда:

- Навыки подключаются только явным конфигом.
- Все пути проходят через политику workspace.
- `ask` и `run` видят одинаковый skill-контекст.
- Нет новых runtime-зависимостей.
- Старые проекты без поля `skills` работают как раньше.
