# Как работает `apply_patch`

`apply_patch` применяет небольшой Codex-style patch внутри `workspace_root`. Инструмент строгий: он не исправляет формат за моделью, сначала проверяет весь patch и только потом пишет файлы. Если один hunk не совпал, workspace остаётся без частичных изменений.

Аргумент `patch` — это одна многострочная строка. Это не shell-команда, не unified diff и не JSON с отдельными полями.

## Общий формат

Patch всегда начинается и заканчивается точными маркерами:

```text
*** Begin Patch
...
*** End Patch
```

Поддерживаются три операции с файлами:

```text
*** Add File: path/to/new.txt
+каждая строка нового файла начинается с плюса
```

```text
*** Update File: path/to/existing.txt
@@
 строка контекста начинается с одного пробела
-удаляемая строка начинается с минуса
+добавляемая строка начинается с плюса
```

```text
*** Delete File: path/to/delete.txt
```

Для перемещения файла используется `Update File`, а `Move to` идёт сразу после него:

```text
*** Update File: old/path.txt
*** Move to: new/path.txt
```

После `Move to` можно добавить обычные `@@` hunks, если файл нужно одновременно переименовать и изменить.

## Примеры

Точечная правка существующего файла:

```text
*** Begin Patch
*** Update File: hello.txt
@@
 one
-two
+deux
 three
*** End Patch
```

Создание файла:

```text
*** Begin Patch
*** Add File: notes/todo.txt
+first task
+second task
*** End Patch
```

Удаление файла:

```text
*** Begin Patch
*** Delete File: old.txt
*** End Patch
```

Перемещение с правкой:

```text
*** Begin Patch
*** Update File: old/name.txt
*** Move to: new/name.txt
@@
-title: old
+title: new
 body
*** End Patch
```

## Частые ошибки

- Нельзя передавать `apply_patch <<'PATCH'` или другую shell-обёртку. В `patch` должен быть только текст от `*** Begin Patch` до `*** End Patch`.
- Нельзя использовать `diff --git`, `---`, `+++` и номера строк unified diff. Этот инструмент ждёт Codex-style patch.
- Если patch не совпал с файлом, сначала перечитайте текущий участок через `read_file` или найдите его через `search_code`, затем повторите `apply_patch` с точным текущим текстом.
- Не переходите к `write_file` для существующего файла после неудачного hunk, если не нужна осознанная полная перезапись файла.
- Не используйте `run_shell` со скриптами для редактирования файлов. `run_shell` нужен для проверок, сборки и безопасной инспекции.
- В `Add File` каждая строка содержимого должна начинаться с `+`.
- В `Update File` hunk должен содержать хотя бы одну строку контекста или удаления. Чистая вставка без привязки к существующему тексту не принимается.
- `*** Move to:` работает только сразу после `*** Update File:`.
- Один и тот же путь нельзя менять двумя отдельными операциями в одном patch.

## Ошибки парсера

| Сообщение | Что исправить |
| --- | --- |
| `patch must start with *** Begin Patch` | Уберите shell-команду, Markdown fence или текст перед первым маркером. |
| `patch must end with *** End Patch` | Поставьте финальный маркер последней строкой patch. |
| `unexpected patch line` | Проверьте заголовок операции или строку hunk: она должна начинаться с пробела, `-`, `+`, `@@` или `***`. |
| `invalid hunk line: ` | В hunk попала пустая строка без маркера. Если это пустая строка контекста, она всё равно должна начинаться с одного пробела. |
| `expected 1 hunk match, found 0` | Контекст или удаляемые строки не совпадают с текущим содержимым файла. Сначала перечитайте точный участок и скопируйте строки verbatim. |
| `expected 1 hunk match, found 2` | Контекст слишком короткий и совпал в нескольких местах. Добавьте соседние строки, скопированные из текущего файла. |
| `update hunk must include context or removed lines` | Добавьте строку контекста или строку, которую нужно удалить. |
| `Move to is only supported after Update File` | Перенесите `*** Move to:` сразу под `*** Update File:`. |
| `outside workspace` или `protected path` | Путь выходит за разрешённую рабочую папку или попадает в защищённую область. |
