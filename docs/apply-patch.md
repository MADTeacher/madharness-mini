# Как работает `apply_patch`

`apply_patch` — инструмент агентского режима для точечных правок файлов без полной перезаписи. Он принимает текст патча, проверяет его целиком и только потом меняет файлы внутри `workspace_root`.

Инструмент написан на чистом Python. Он не вызывает `git apply`, `patch`, shell-команды или внешние зависимости.

## Зачем он нужен

`write_file` удобен для новых файлов, но опасен для маленьких правок: модель должна передать весь файл целиком и может случайно потерять остальной текст.

`apply_patch` меняет только описанные строки. Если контекст не найден или найден неоднозначно, инструмент возвращает ошибку и ничего не записывает.

Для простой замены одной известной строки лучше использовать `replace_text`. Для связанных многострочных изменений удобнее `apply_patch`.

## Формат патча

Патч начинается и заканчивается служебными строками:

```patch
*** Begin Patch
*** Update File: example.txt
@@
 old line
-line to remove
+line to add
*** End Patch
```

Поддерживаются три операции:

| Операция | Что делает |
| --- | --- |
| `*** Add File: path` | Создаёт новый файл. Все строки содержимого начинаются с `+`. |
| `*** Update File: path` | Меняет существующий файл по одному или нескольким hunk-блокам. |
| `*** Delete File: path` | Удаляет существующий файл. |

Внутри `Update File` строки hunk-блока имеют простой смысл:

| Префикс | Смысл |
| --- | --- |
| пробел | Контекстная строка, должна совпасть с файлом. |
| `-` | Строка, которую нужно удалить. |
| `+` | Строка, которую нужно добавить. |

Строка `@@` отделяет hunk-блоки. Сейчас координаты строк после `@@` не используются: инструмент ищет точный текст старого блока в текущем файле.

## Перемещение файла

`apply_patch` поддерживает `*** Move to:` только внутри `Update File`:

```patch
*** Begin Patch
*** Update File: docs/old.md
*** Move to: examples/new.md
@@
-old title
+new title
*** End Patch
```

Если hunk-блоков нет, это обычное перемещение файла:

```patch
*** Begin Patch
*** Update File: docs/old.md
*** Move to: docs/new.md
*** End Patch
```

Перемещать можно только файлы внутри workspace. Целевой файл не должен существовать.

## Безопасность и ошибки

Все пути проходят через `Policy.safe_path()`. Это значит, что нельзя выйти за `workspace_root` или писать в защищённые пути вроде `.git`, `.env`, `secrets` и `~/.ssh`.

Перед записью инструмент готовит все изменения в памяти:

1. Разбирает формат патча.
2. Проверяет пути.
3. Проверяет, что файлы существуют или не существуют там, где нужно.
4. Для `Update File` ищет старый hunk в текущем содержимом.
5. Если любой шаг падает, возвращает `ok: false` и не меняет файлы.

Для hunk-блока требуется ровно одно совпадение старого текста. Если совпадений нет или их несколько, патч считается небезопасным:

```text
expected 1 hunk match, found 0
expected 1 hunk match, found 2
```

Такой отказ лучше, чем случайная правка не того места.

## Что не поддерживается

`apply_patch` намеренно остаётся маленьким:

- не поддерживает перемещение директорий;
- не перезаписывает существующий целевой файл при `Move to`;
- не применяет fuzzy matching;
- не понимает полный формат `git diff`;
- не выполняет shell-команды.

Если нужен сложный рефакторинг, агент должен сначала прочитать нужные файлы, затем сделать несколько небольших патчей и после этого запустить проверку.
