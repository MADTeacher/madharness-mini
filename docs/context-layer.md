# Слой контекста

Слой контекста отвечает за то, какие сообщения модель видит в `ask` и `run`.
Он живёт в пакете `madharness_mini.context` и не зависит от конфига, политики
безопасности, model client или handlers инструментов. Внешний код передаёт ему
готовые тексты и события, а на выходе получает список `messages` для
OpenAI-совместимого Chat Completions API.

`messages` — это ровно тот список сообщений, который харнесс отправляет в поле
`messages` HTTP-запроса к модели: системные правила, задача пользователя,
ответы assistant и результаты инструментов.

## Основные типы

`ContextFragment` описывает отдельный кусок контекста: системный промпт,
проектные инструкции, подсказку навыка или будущий MCP-фрагмент. У фрагмента есть
`id`, `source`, `text`, `priority`, `placement` и `transient`.

`ContextProvider` — простой protocol для расширений. Provider получает
`ContextState` и возвращает дополнительные `ContextFragment`. Так будущие
навыки, MCP, субагенты и команды смогут добавлять инструкции без импорта
`loop.py` или `model.py`.

`ContextManager` хранит задачу пользователя, закреплённые фрагменты и историю
assistant/tool сообщений. Основной метод `messages()` возвращает готовый список
сообщений для модели, а `stats()` показывает примерный размер собранного
контекста и факт обрезки.

## Как используется в `run`

`loop.py` создаёт `ContextManager` через `base_context()`, читает встроенный
промпт и `AGENTS.md`, затем добавляет их как закреплённый `ContextFragment`.

Закреплённый фрагмент — это часть контекста, которую нельзя удалить при обрезке
старой истории. Например, системный промпт и правила проекта должны оставаться
видимыми модели даже тогда, когда результаты старых инструментов уже не влезают
в бюджет контекста.

После каждого ответа модели `run_agent()` вызывает `record_assistant()`. Если
модель запросила инструмент, результат записывается через `record_tool_result()`.
Сообщение assistant с `tool_calls` и соответствующие `role=tool` ответы хранятся
как атомарный элемент истории: при обрезке они сохраняются или удаляются вместе.

Мультимодальные follow-up сообщения, например вложение из `read_image`,
добавляются после закрытия всех `tool_calls`. Observation инструмента остаётся
текстовым JSON без base64, поэтому трасса не раздувается изображениями.

## Бюджет контекста

В V1 харнесс ограничивает контекст простым и предсказуемым способом: смотрит на
примерную длину JSON-представления `messages` в символах. Если история стала
слишком длинной, старые результаты инструментов сокращаются или удаляются.

Настройки:

| Поле | Значение по умолчанию | Смысл |
| --- | --- | --- |
| `context_max_chars` | `120000` | Примерный максимум символов в истории, отправляемой модели. |
| `context_keep_recent_turns` | `3` | Сколько последних элементов истории сохранять при удалении старых. |

Закреплённые фрагменты и исходная задача пользователя не удаляются. Если контекст
слишком большой, сначала укорачиваются `role=tool` outputs с явной пометкой
`context clipped`. Если этого мало, старые неприкреплённые элементы истории
удаляются целиком.

## Расширение инструментов и контекста

Инструменты расширяются через `ToolProvider`, а контекст — через
`ContextProvider`. Для будущего навыка или MCP-сервера это означает две
независимые точки подключения:

- `ToolProvider.specs(ctx)` отдаёт `ToolSpec` для вызова модели.
- `ContextProvider.collect(state)` отдаёт инструкции или другие фрагменты,
  которые модель должна видеть перед вызовом инструмента.

Так инструмент может появиться без правки агентского цикла, а контекстные
инструкции не смешиваются с кодом выполнения handlers.
