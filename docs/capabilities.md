# Возможности харнесса

`madharness-mini` помогает модели работать с локальным репозиторием. Всё лежит рядом с проектом. Настройки находятся в `.madharness-mini/`, а каждый запуск можно разобрать по трассе.

## Режимы работы

| Команда | Что делает | Когда использовать |
| --- | --- | --- |
| `madharness-mini ask "..."` | Отправляет вопрос модели без инструментов. | Нужно короткое объяснение или проверка доступа к модели. |
| `madharness-mini run "..."` | Запускает агента с инструментами для папки проекта. | Нужно читать файлы, искать по проекту, менять код или запускать проверку. |
| `madharness-mini init` | Создаёт настройку в `.madharness-mini/config.json`. | Проект ещё не подключён к `madharness-mini`. |
| `madharness-mini trace <идентификатор>` | Показывает краткую сводку по журналу запуска. | Нужно понять, какие шаги были в сессии. |
| `madharness-mini skills list/show/validate` | Показывает найденные Agent Skills и диагностику `SKILL.md`. | Нужно проверить, какие проектные навыки увидит `run`. |

`ask` подходит для простого вопроса. Модель получает встроенный системный промпт и ваш текст, потом сразу отвечает.

`run` работает по шагам. Модель может ответить сразу или попросить вызвать инструмент. Харнесс вызывает инструмент, возвращает результат модели, и работа продолжается до ответа или до лимита ходов.

## Проектные инструкции `AGENTS.md`

Перед первым обращением к модели `madharness-mini` добавляет к встроенному системному промпту проектные инструкции из файлов `AGENTS.md`.

Поиск начинается в `workspace_root` и идёт вниз до активной папки запуска, если она находится внутри `workspace_root`. Файлы соединяются в этом порядке: сначала правила корня проекта, затем более вложенные правила. Пустые файлы пропускаются.

Поддерживается только точное имя `AGENTS.md`. Глобальные файлы из домашнего каталога, `AGENTS.override.md` и fallback-имена не читаются. Общий объём добавленных проектных инструкций ограничен 32 KiB, как значение по умолчанию в Codex.

## Agent Skills

В режиме `run` харнесс поддерживает project-local Agent Skills без отдельной настройки. Discovery смотрит прямые подпапки двух каталогов:

```text
.madharness_mini/skills
.agents/skills
```

Каждый skill — это папка с `SKILL.md`. Обязательные поля `name` и `description` читаются из YAML frontmatter, а тело markdown добавляется в контекст только после активации. Поддерживаются optional-поля `license`, `compatibility`, `metadata` и experimental `allowed-tools`.

Явный выбор работает через маркеры `@skill:name`, `@skill/name`, `$name` и фразы вида `используй навык name`. Если такой skill не найден, `run` завершается до обращения к модели с понятной ошибкой. Когда явного выбора нет, модель видит компактный catalog найденных навыков и может вызвать `activate_skill`, чтобы подключить нужный workflow. `ask` skill-каталог не загружает и маркеры не разбирает.

Bundled resources (`scripts/`, `references/`, `assets/` и другие файлы внутри skill root) перечисляются при активации, но не читаются заранее. Модель может прочитать конкретный файл через обычный `read_file` по workspace-relative пути или запустить documented script через `run_shell` с `cwd`, если команда проходит общую shell-политику.

Подробно принцип работы навыков описан в документе [Agent Skills в madharness-mini](agent-skills.md).

## MCP tools

В режиме `run` харнесс может подключать tools внешних stdio MCP-серверов. Настройки лежат отдельно от основного конфига:

```text
.madharness-mini/mcp.json
```

Если этого файла нет, MCP выключен и набор инструментов остаётся прежним. Минимальный формат:

```json
{
  "servers": {
    "playwright": {
      "enabled": true,
      "command": "npx",
      "args": ["-y", "@playwright/mcp@latest"],
      "cwd": ".",
      "timeout_seconds": 30
    }
  }
}
```

Поддерживается только stdio transport и только MCP tools. После `initialize` харнесс вызывает `tools/list`, превращает каждый MCP tool в обычный `ToolSpec` с именем вида `mcp__playwright__browser_navigate`, а при вызове модели отправляет на сервер `tools/call`.

MCP-сервер запускается как внешний процесс без shell-строки: `command` и `args` не склеиваются. `cwd` обязан оставаться внутри `workspace_root`. Окружение сервера состоит из безопасного минимума системных переменных и явного `env` из `mcp.json`; ключ LLM API и `MADHARNESS_MINI_*` не передаются автоматически.

## Слой контекста

Перед каждым обращением к модели `madharness-mini` собирает контекст через `ContextManager` из пакета `madharness_mini.context`.

В контекст входят закреплённые фрагменты (системный промпт и проектные инструкции), исходная задача пользователя и история ответов модели с результатами инструментов. Закреплённые фрагменты нельзя удалить при обрезке старой истории. Assistant-сообщение с `tool_calls` и соответствующие `role=tool` observations хранятся как атомарная пара: если старую историю нужно удалить из-за бюджета, вызов инструмента и его результат не разрываются.

Контекст ограничивается по примерной токеновой оценке компактного JSON-запроса. В режиме `run` в оценку входят и `messages`, и схемы доступных `tools`. Assistant-ответы сохраняются без vendor-specific полей вроде `reasoning_details`, а длинный assistant text укорачивается. Сначала укорачиваются большие tool outputs с пометкой `context clipped`, затем удаляются старые элементы истории. Если недавняя история всё равно не помещается, `context_keep_recent_turns` уступает жёсткому лимиту; если даже пустая история со схемами tools больше лимита, запуск завершается ошибкой бюджета вместо слишком большого запроса к модели.

Каждый вызов модели пишет в трассу `context_report`: короткую диагностику состава контекста, размеров фрагментов, обрезанных tool messages и удалённых элементов истории. Команда `trace` показывает последнюю сводку одной строкой, чтобы было проще понять, почему модель видела именно такой контекст.

Подробнее устройство слоя описано в документе [Слой контекста](context-layer.md).

## Инструменты агента

| Инструмент | Что делает |
| --- | --- |
| `list_files` | Рекурсивно показывает файлы внутри рабочей папки: только файлы, с glob-фильтром по имени и лимитом 200 результатов. |
| `read_file` | Читает UTF-8 фрагмент файла по 1-based строкам и возвращает текст с номерами строк. |
| `read_image` | Проверяет локальное PNG, JPEG, WEBP или неанимированное GIF и при включённом vision-вводе прикрепляет изображение к следующему запросу модели. |
| `search_code` | Ищет буквальную подстроку в файлах проекта: не regex и не семантический поиск, до 100 совпадений. |
| `apply_patch` | Применяет небольшой патч для точечных правок, создания, удаления или перемещения файлов. |
| `write_file` | Записывает полный UTF-8 текст файла, создаёт родительские каталоги и полностью перезаписывает цель. |
| `run_shell` | Запускает одну разрешённую команду в рабочей папке или безопасном workspace-relative `cwd`; управляющие операторы shell запрещены. |
| `activate_skill` | Появляется только в `run`, когда найдены skills и пользователь не выбрал skill явно; добавляет инструкции навыка в durable context. |

Обычно агент идёт таким путём: находит нужный файл, читает его, делает правку, запускает проверку. Если инструмент вернул ошибку, агент должен учитывать эту ошибку.

Подробнее формат патчей описан в документе [Как работает `apply_patch`](apply-patch.md).

Для разработки новых встроенных инструментов используется пакет `madharness_mini.tools`. Handler и `ToolSpec` добавляются в подходящий модуль пакета, а стандартный набор отдаёт `BuiltinToolProvider`. Skills и MCP подключаются через дополнительные providers, поэтому внешний источник может добавить инструмент без правки агентского цикла.

## Настройки

Основной файл настроек появляется после команды `init`:

```text
.madharness-mini/config.json
```

Главные поля:

| Поле | Смысл |
| --- | --- |
| `model` | Модель, которую нужно вызвать. По умолчанию задана `deepseek/deepseek-v4-flash`. |
| `base_url` | Адрес API, куда отправляется запрос к модели. |
| `api_key` | Ключ доступа API. Это секретная строка для выбранного сервиса. |
| `temperature` | Насколько свободно модель выбирает ответ. Меньше значение - спокойнее и предсказуемее ответ. |
| `max_turns` | Максимальное число ходов в режиме `run`. |
| `context_max_tokens` | Примерный максимум токенов в запросе, который отправляется модели. |
| `context_keep_recent_turns` | Сколько последних элементов истории сохранять при бюджетной обрезке. |
| `workspace_root` | Папка, внутри которой агент может работать. |
| `protected_paths` | Пути, к которым инструменты не должны обращаться. |
| `allow_shell` | Разрешает или запрещает инструмент `run_shell`. |

MCP не добавляет поле в этот файл. Для MCP используется отдельный `.madharness-mini/mcp.json`.

Настройки можно переопределить через файл `.env` в корне проекта или через переменные окружения:

| Переменная | Какое поле меняет |
| --- | --- |
| `MADHARNESS_MINI_MODEL` | `model` |
| `MADHARNESS_MINI_BASE_URL` | `base_url` |
| `MADHARNESS_MINI_API_KEY` | `api_key` |

Файл `.env` удобен для локальных настроек. Секреты лучше не добавлять в Git.

## Адрес LLM API

Сервис маршрутизации выбирается одним полем: `base_url`.

По умолчанию `madharness-mini` отправляет запросы на OpenRouter:

```text
https://openrouter.ai/api/v1
```

Если нужен другой сервис, задайте его адрес через `--base-url`, `MADHARNESS_MINI_BASE_URL` или ручную правку `.madharness-mini/config.json`.

Готовые значения:

| Сервис | Адрес API |
| --- | --- |
| OpenRouter | `https://openrouter.ai/api/v1` |
| KodikRouter | `https://kodikrouter.ru/api/v1` |
| Локальный OpenAI-совместимый сервер | `http://127.0.0.1:11434/v1` |
| Свой адрес | Любой OpenAI-совместимый адрес с путём `/v1` |

При первой настройке достаточно указать модель и ключ:

```bash
madharness-mini init \
  --model deepseek/deepseek-v4-flash \
  --api-key "ключ-доступа-openrouter"
```

Для KodikRouter команда будет такой:

```bash
madharness-mini init \
  --base-url https://kodikrouter.ru/api/v1 \
  --model deepseek/deepseek-v4-flash \
  --api-key "ключ-доступа-kodikrouter"
```

Ключ должен подходить выбранному сервису. Для OpenRouter нужен ключ OpenRouter. Для KodikRouter нужен ключ KodikRouter. Локальный сервис может работать без ключа, если так настроен его API.

## Лимиты LLM API

Если провайдер возвращает HTTP 429, это обычно означает, что достигнут лимит запросов, токенов или баланса для выбранной модели/ключа.

Когда ответ содержит короткий заголовок `Retry-After`, `madharness-mini` ждёт до 60 секунд и один раз повторяет запрос. Если заголовка нет, ожидание слишком длинное или повтор снова получает 429, запуск завершается понятной ошибкой. В таком случае попробуйте позже, смените модель или проверьте лимиты провайдера.

## Трассы выполнения

Каждая команда `ask` и `run` создаёт файл трассы:

```text
.madharness-mini/traces/<дата>-<время>-<суффикс>.jsonl
```

Трасса - это журнал запуска, на каждой строке которого лежит одна JSON-запись.

В трассе есть старт сессии, вызовы модели, ответы инструментов и итоговый результат.
У событий `model_call_started` есть `context_report` — отчёт о примерном
токеновом размере запроса, в том числе схем инструментов, и о том, что было
обрезано бюджетом.

Для Agent Skills дополнительно пишутся события `skills_discovered`, `skill_activated` и `skill_resource_used`. Трасса не дублирует полный текст активированного `SKILL.md`, чтобы журнал не раздувался и не раскрывал лишний контент.

Для MCP дополнительно пишутся `mcp_server_started`, `mcp_server_error` и `mcp_server_stopped`. В эти события не попадают секреты и полный вывод stderr/stdout.

Краткая сводка открывается командой:

```bash
madharness-mini trace <начало-имени-файла>
```

Она помогает понять, где задача остановилась, какие инструменты вызывались и что вернула модель.

## Ограничения безопасности

Агент работает внутри папки `workspace_root`. Попытка прочитать или записать файл снаружи этой папки завершится ошибкой.

Защищённые пути задаются в `protected_paths`. По умолчанию туда входят `.git`, `.env`, `secrets` и `~/.ssh`.

Инструмент `run_shell` проверяет команду перед запуском. Он запрещает рискованные фрагменты вроде `rm -rf`, `sudo`, `curl`, `wget`, `ssh`, `scp`, `chmod 777`, `mkfs`, `dd`.

Управляющие операторы оболочки запрещены: `|`, `>`, `<`, `&&`, `||`, `;`. Команда должна быть одной простой командой с аргументами.

## Типичный рабочий сценарий

1. Пользователь описывает задачу обычным языком.
2. Агент ищет файлы и читает нужные фрагменты.
3. Агент вносит точечные изменения через `apply_patch`.
4. Агент запускает доступную проверку, если она подходит задаче.
5. Пользователь получает короткий итог и путь к трассе.

Такой сценарий удобен для учебного проекта. Видно, что сделал агент, где он ошибся, какой инструмент помог, а какой вернул отказ.
