# Возможности харнесса

`madharness-mini` помогает модели работать с локальным репозиторием. Всё лежит рядом с проектом. Настройки находятся в `.madharness-mini/`, а каждый запуск можно разобрать по трассе.

## Режимы работы

| Команда | Что делает | Когда использовать |
| --- | --- | --- |
| `madharness-mini ask "..."` | Отправляет вопрос модели без инструментов. | Нужно короткое объяснение или проверка доступа к модели. |
| `madharness-mini run "..."` | Запускает агента с инструментами для папки проекта. | Нужно читать файлы, искать по проекту, менять код или запускать проверку. |
| `madharness-mini init` | Создаёт настройку в `.madharness-mini/config.json`. | Проект ещё не подключён к `madharness-mini`. |
| `madharness-mini trace <идентификатор>` | Показывает краткую сводку по журналу запуска. | Нужно понять, какие шаги были в сессии. |

`ask` подходит для простого вопроса. Модель получает встроенный системный промпт и ваш текст, потом сразу отвечает.

`run` работает по шагам. Модель может ответить сразу или попросить вызвать инструмент. Харнесс вызывает инструмент, возвращает результат модели, и работа продолжается до ответа или до лимита ходов.

## Проектные инструкции `AGENTS.md`

Перед первым обращением к модели `madharness-mini` добавляет к встроенному системному промпту проектные инструкции из файлов `AGENTS.md`.

Поиск начинается в `workspace_root` и идёт вниз до активной папки запуска, если она находится внутри `workspace_root`. Файлы соединяются в этом порядке: сначала правила корня проекта, затем более вложенные правила. Пустые файлы пропускаются.

Поддерживается только точное имя `AGENTS.md`. Глобальные файлы из домашнего каталога, `AGENTS.override.md` и fallback-имена не читаются. Общий объём добавленных проектных инструкций ограничен 32 KiB, как значение по умолчанию в Codex.

## Инструменты агента

| Инструмент | Что делает |
| --- | --- |
| `list_files` | Показывает файлы внутри рабочей папки. |
| `read_file` | Читает нужный кусок файла по строкам. |
| `search_code` | Ищет точный текст в файлах проекта. |
| `replace_text` | Заменяет точный текст в существующем файле. |
| `apply_patch` | Применяет небольшой патч для точечных правок, создания, удаления или перемещения файлов. |
| `write_file` | Записывает текст UTF-8 в новый файл или полностью перезаписывает существующий файл. |
| `run_shell` | Запускает разрешённую команду в рабочей папке. |

Обычно агент идёт таким путём: находит нужный файл, читает его, делает правку, запускает проверку. Если инструмент вернул ошибку, агент должен учитывать эту ошибку.

Подробнее формат патчей описан в документе [Как работает `apply_patch`](apply-patch.md).

Для разработки новых встроенных инструментов используется пакет `madharness_mini.tools`. Handler и `ToolSpec` добавляются в подходящий модуль пакета, затем spec подключается в явный список `builtin_specs()` в `builtins.py`.

## Настройки

Основной файл настроек появляется после команды `init`:

```text
.madharness-mini/config.json
```

Главные поля:

| Поле | Смысл |
| --- | --- |
| `model` | Модель, которую нужно вызвать. По умолчанию задана `deepseek/deepseek-v4-flash`. |
| `base_url` | Адрес API, куда отправляется запрос к модели. |
| `api_key` | Ключ доступа API. Это секретная строка для выбранного сервиса. |
| `temperature` | Насколько свободно модель выбирает ответ. Меньше значение - спокойнее и предсказуемее ответ. |
| `max_turns` | Максимальное число ходов в режиме `run`. |
| `workspace_root` | Папка, внутри которой агент может работать. |
| `protected_paths` | Пути, к которым инструменты не должны обращаться. |
| `allow_shell` | Разрешает или запрещает инструмент `run_shell`. |

Настройки можно переопределить через файл `.env` в корне проекта или через переменные окружения:

| Переменная | Какое поле меняет |
| --- | --- |
| `MADHARNESS_MINI_MODEL` | `model` |
| `MADHARNESS_MINI_BASE_URL` | `base_url` |
| `MADHARNESS_MINI_API_KEY` | `api_key` |

Файл `.env` удобен для локальных настроек. Секреты лучше не добавлять в Git.

## Адрес LLM API

Сервис маршрутизации выбирается одним полем: `base_url`.

По умолчанию `madharness-mini` отправляет запросы на OpenRouter:

```text
https://openrouter.ai/api/v1
```

Если нужен другой сервис, задайте его адрес через `--base-url`, `MADHARNESS_MINI_BASE_URL` или ручную правку `.madharness-mini/config.json`.

Готовые значения:

| Сервис | Адрес API |
| --- | --- |
| OpenRouter | `https://openrouter.ai/api/v1` |
| KodikRouter | `https://kodikrouter.ru/api/v1` |
| Локальный OpenAI-совместимый сервер | `http://127.0.0.1:11434/v1` |
| Свой адрес | Любой OpenAI-совместимый адрес с путём `/v1` |

При первой настройке достаточно указать модель и ключ:

```bash
madharness-mini init \
  --model deepseek/deepseek-v4-flash \
  --api-key "ключ-доступа-openrouter"
```

Для KodikRouter команда будет такой:

```bash
madharness-mini init \
  --base-url https://kodikrouter.ru/api/v1 \
  --model deepseek/deepseek-v4-flash \
  --api-key "ключ-доступа-kodikrouter"
```

Ключ должен подходить выбранному сервису. Для OpenRouter нужен ключ OpenRouter. Для KodikRouter нужен ключ KodikRouter. Локальный сервис может работать без ключа, если так настроен его API.

## Лимиты LLM API

Если провайдер возвращает HTTP 429, это обычно означает, что достигнут лимит запросов, токенов или баланса для выбранной модели/ключа.

Когда ответ содержит короткий заголовок `Retry-After`, `madharness-mini` ждёт до 60 секунд и один раз повторяет запрос. Если заголовка нет, ожидание слишком длинное или повтор снова получает 429, запуск завершается понятной ошибкой. В таком случае попробуйте позже, смените модель или проверьте лимиты провайдера.

## Трассы выполнения

Каждая команда `ask` и `run` создаёт файл трассы:

```text
.madharness-mini/traces/<дата>-<время>-<суффикс>.jsonl
```

Трасса - это журнал запуска, на каждой строке которого лежит одна JSON-запись.

В трассе есть старт сессии, вызовы модели, ответы инструментов и итоговый результат.

Краткая сводка открывается командой:

```bash
madharness-mini trace <начало-имени-файла>
```

Она помогает понять, где задача остановилась, какие инструменты вызывались и что вернула модель.

## Ограничения безопасности

Агент работает внутри папки `workspace_root`. Попытка прочитать или записать файл снаружи этой папки завершится ошибкой.

Защищённые пути задаются в `protected_paths`. По умолчанию туда входят `.git`, `.env`, `secrets` и `~/.ssh`.

Инструмент `run_shell` проверяет команду перед запуском. Он запрещает рискованные фрагменты вроде `rm -rf`, `sudo`, `curl`, `wget`, `ssh`, `scp`, `chmod 777`, `mkfs`, `dd`.

Управляющие операторы оболочки запрещены: `|`, `>`, `<`, `&&`, `||`, `;`. Команда должна быть одной простой командой с аргументами.

## Типичный рабочий сценарий

1. Пользователь описывает задачу обычным языком.
2. Агент ищет файлы и читает нужные фрагменты.
3. Агент вносит точечные изменения через `replace_text` или `apply_patch`.
4. Агент запускает доступную проверку, если она подходит задаче.
5. Пользователь получает короткий итог и путь к трассе.

Такой сценарий удобен для учебного проекта. Видно, что сделал агент, где он ошибся, какой инструмент помог, а какой вернул отказ.
