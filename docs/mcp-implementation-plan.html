Архитектурная карта

MCP не должен становиться отдельным агентским циклом. Он добавляет внешний источник инструментов, а дальше пользуется тем же путём, что и встроенные tools и Agent Skills.

MCP server

Локальный процесс из .madharness-mini/mcp.json: получает JSON-RPC по stdin, отвечает по stdout.

→

mcp/stdio.py

Запускает subprocess, делает initialize, tools/list, tools/call, закрывает процесс.

→

McpToolProvider

Преобразует MCP tools в ToolSpec с безопасным префиксом имени.

ToolRegistry

Регистрирует MCP specs рядом со встроенными tools и вызывает handler по имени.

→

ModelClient

Отправляет модели обычные OpenAI-compatible function schemas.

→

Observation

tools/call возвращается в формат ok() или fail() и уходит в историю контекста.

План внедрения

Протокольное ядро

  • Ввести константу MCP_PROTOCOL_VERSION = "2025-11-25".
  • Собрать helpers для request, notification и response parsing.
  • На ошибочный JSON-RPC response поднимать понятный RuntimeError.
низкий риск

Stdio transport

  • Запускать сервер через subprocess.Popen([...], shell=False).
  • Читать stdout и stderr раздельно, stdout считать только protocol channel.
  • Поддержать timeout на initialize, list и call.
процессы

Конфиг и safety

  • Читать MCP-настройки из .madharness-mini/mcp.json; если файла нет, MCP выключен.
  • Принимать только явно включённые серверы: enabled: true.
  • Проверять cwd через workspace policy и не передавать модельный api_key в env сервера.
важно

Tool adapter

  • Сделать McpToolProvider.specs(ctx).
  • Префиксовать имена как mcp__<server>__<tool>.
  • Хранить исходное имя MCP tool внутри handler для tools/call.
ядро готово

Result conversion

  • Текстовые content items объединять в content.
  • Image, audio, resource и resource links возвращать как metadata без base64 payload.
  • isError: true превращать в fail(), даже если JSON-RPC response успешен.
контекст

Lifecycle в run

  • Создавать MCP providers перед ToolRegistry.
  • Добавить ToolRegistry.close().
  • Обернуть агентский цикл в try/finally, чтобы процессы закрывались при финале и при ошибках.
обязателен

Trace, docs, regression

  • Писать короткие MCP события без секретов и без полного stderr/stdout.
  • Добавить fake MCP server в tests для lifecycle и tool calls.
  • Обновить документацию и прогнать uv run -m unittest discover -s tests.
закрытие V1

Конфиг V1

Формат живёт в отдельном файле .madharness-mini/mcp.json. Если файла нет, MCP полностью выключен без изменений в defaults. Команда и аргументы разделены, чтобы не строить shell-строку и не расширять текущую shell-политику. 

{
  "servers": {
    "demo": {
      "enabled": true,
      "command": "python3",
      "args": ["scripts/demo_mcp_server.py"],
      "cwd": ".",
      "env": {
        "DEMO_MODE": "1"
      },
      "timeout_seconds": 20
    }
  }
}

Минимальный JSON-RPC путь

V1 поддерживает только три рабочие операции после запуска процесса: инициализация, список инструментов и вызов конкретного инструмента.

  • initialize: версия протокола, capabilities клиента, clientInfo.
  • notifications/initialized: переход в нормальную работу.
  • tools/list: получение name, description, inputSchema.
  • tools/call: отправка исходного имени tool и arguments.
  • Shutdown: close stdin, wait, terminate, kill при зависании.

Границы V1

  • Не добавлять официальный MCP SDK.
  • Не добавлять runtime-зависимости в pyproject.toml.
  • Не поддерживать Streamable HTTP в первой версии.
  • Не поддерживать resources, prompts, sampling, roots, elicitation, progress и tasks.
  • Не делать auto-discovery серверов вне явного конфига.

Открытые решения

  • Нужна ли allowlist команд MCP-серверов сверх явного enabled: true?
  • Должен ли env наследовать минимальный системный PATH или быть полностью явным?
  • Принимать ли совместимые старые версии протокола или строго только 2025-11-25?
  • Как показать пользователю warning перед первым запуском MCP-сервера: документацией, trace-событием или CLI-подтверждением?
Рекомендуемое начальное решение: strict version, явный конфиг, shell=False, workspace-relative cwd, env без модельного api_key, короткая диагностика stderr только в trace.

Критерии готовности

  • Один stdio MCP-сервер отдаёт tools через tools/list.
  • Модель видит MCP tools в общем списке schemas.
  • Вызов mcp__server__tool проходит через ToolRegistry.call().
  • Ответ MCP tool превращается в обычный observation.
  • Процесс MCP-сервера закрывается после run_agent() при успехе и ошибке.
  • Старые проекты без .madharness-mini/mcp.json работают без изменений.
  • В trace видны старт, сбой и остановка MCP-сервера без секретов.
  • Полный unittest suite проходит.
