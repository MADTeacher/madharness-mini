# Лабораторные работы: 07-hooks

Эта ветка тренирует lifecycle hooks: проектный код может наблюдать события
harness, писать аудит и блокировать `before_tool_call` до выполнения действия.
Главная инженерная граница: hooks расширяют политику проекта, но не заменяют
базовую `Policy`.

## Базовый уровень

### 1. Проверить загрузку пустого `hooks.json`

Цель: понять, как hook manager ведёт себя без обработчиков.

Задание: добавьте тест, где `.madharness-mini/hooks.json` существует, но список
hooks пустой.

Ограничения:

- запуск не должен падать;
- trace не должен содержать ложных `hook_started`;
- поведение должно совпадать с отсутствующим файлом.

Критерий готовности: пустой конфиг является валидным no-op.

### 2. Заблокировать `run_shell` через `before_tool_call`

Цель: увидеть главный сценарий hooks — отказ до handler-а инструмента.

Задание: добавьте fixture hook-команду, которая блокирует `run_shell`, и тест
на observation `ok=false`.

Ограничения:

- handler `run_shell` не должен запускаться;
- модель должна получить обычный tool result;
- trace должен содержать `hook_blocked`.

Критерий готовности: блокировка выглядит для model loop как штатный fail tool
result.

### 3. Проверить redaction секретов

Цель: понять, что hook получает payload, но не должен видеть очевидные секреты.

Задание: добавьте тест на поля вроде `api_key`, `password`, `secret` или
`*_token` в данных события.

Ограничения:

- redaction не должен ломать форму JSON;
- обычные поля события должны сохраниться;
- тест не должен использовать реальный секрет пользователя.

Критерий готовности: hook payload содержит `<redacted>` вместо секретного
значения.

## Основной уровень

### 1. Добавить match по списку значений

Цель: сделать routing hooks удобнее без языка правил.

Задание: проверьте или реализуйте `match`, где значение может быть списком:
например `{ "tool": ["write_file", "apply_patch"] }`.

Ограничения:

- exact-match для строк должен остаться прежним;
- списки не должны превращаться в regex;
- тест должен покрыть совпадение и несовпадение.

Критерий готовности: один hook можно применить к нескольким инструментам.

### 2. Сделать ошибки hook-команд наблюдаемыми

Цель: понять, почему audit hooks не должны ломать весь запуск.

Задание: добавьте тест, где hook завершается с non-zero code или печатает
невалидный JSON.

Ограничения:

- запуск должен продолжиться, кроме валидной блокировки `before_tool_call`;
- trace должен содержать `hook_failed`;
- stderr hook-а должен быть обрезан по лимиту.

Критерий готовности: пользователь видит проблему hook-а, но агентский запуск не
падает без необходимости.

### 3. Добавить hook только для субагентов

Цель: связать hooks с оркестрацией.

Задание: добавьте тестовый hook с `match: { "kind": "subagent" }`, который
логирует события только дочерних запусков.

Ограничения:

- parent `run` не должен матчиться;
- дочерний trace должен получить hook events;
- блокировка в субагенте должна работать так же, как в parent.

Критерий готовности: по trace видно, что hooks различают `run` и `subagent`.

## Продвинутый уровень

### 1. Написать guard hook для защищённого файла

Цель: применить hooks как проектную политику поверх общей `Policy`.

Задание: добавьте пример hook-а, который запрещает читать `data/secret.txt` и
запрещает shell-команды удаления.

Ограничения:

- не использовать shell для запуска hook-а;
- hook должен возвращать валидный JSON;
- пример должен быть маленьким и документированным.

Критерий готовности: guard hook демонстрирует policy overlay без изменения
ядра harness.

### 2. Добавить событие для `tool_call` после блокировки

Цель: улучшить наблюдаемость блокировки.

Задание: расширьте trace так, чтобы после `hook_blocked` было легко увидеть,
какое observation получила модель.

Ограничения:

- не вводить новый протокол для модели;
- не дублировать большие args;
- тест должен проверять порядок событий.

Критерий готовности: человек по trace видит и причину блокировки, и итоговый
tool result.

### 3. Спроектировать hook-аудит для учебного проекта

Цель: собрать практичный набор hooks для лабораторной среды.

Задание: создайте пример audit hook, который пишет компактный JSONL-журнал всех
`apply_patch` и `run_shell` событий в локальный файл проекта.

Ограничения:

- не писать секреты;
- не ломать запуск при ошибке audit-файла;
- не коммитить generated log;
- docs должны объяснить, как включить пример.

Критерий готовности: преподаватель может включить audit и увидеть, какие
действия выполнял агент в лабораторной.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

После выполнения задачи проверьте:

- какой event вызвал hook;
- какой `kind` был у события;
- увидел ли hook redacted payload;
- был ли handler инструмента реально запущен;
- что получила модель как observation.
