# Лабораторные работы: 04-Agents-Skills

Эта ветка тренирует Agent Skills как учебный механизм расширения harness:
локальный `SKILL.md` не является кодом инструмента, но меняет контекст, workflow
и поведение модели в `run`.

## Базовый уровень

### 1. Сделать ошибку невалидного `SKILL.md` точнее

Цель: разобраться в loader-е skills и диагностике frontmatter.

Задание: улучшите сообщение для skill без `name` или `description`.

Ограничения:

- `skills validate` должен показывать путь к проблемному файлу;
- невалидный skill не должен попадать в catalog;
- тест должен покрыть хотя бы один error и один warning.

Критерий готовности: студент по выводу `skills validate` понимает, какой файл
исправлять и почему.

### 2. Проверить приоритет каталогов skills

Цель: понять, как harness выбирает skill, если одинаковое имя найдено в двух
местах.

Задание: добавьте тест на конфликт `.agents/skills/<name>` и
`.madharness_mini/skills/<name>`.

Ограничения:

- порядок приоритета должен быть явно описан в docs;
- диагностика конфликта не должна ломать валидный skill;
- не менять имена каталогов.

Критерий готовности: поведение при конфликте стабильно и объяснено.

### 3. Улучшить вывод `skills show`

Цель: увидеть, какие данные skill отдаёт человеку и модели.

Задание: сделайте вывод ресурсов skill более удобным: путь, тип и размер должны
читаться с первого взгляда.

Ограничения:

- не читать содержимое ресурсов автоматически;
- не показывать файлы за пределами skill root;
- покрыть формат тестом CLI.

Критерий готовности: `skills show <name>` помогает понять, что skill приносит
с собой помимо markdown-инструкций.

## Основной уровень

### 1. Сжать catalog skills для модели

Цель: почувствовать, что catalog тоже занимает бюджет контекста.

Задание: ограничьте длину описаний skills в compact catalog и добавьте пометку,
если описание было обрезано.

Ограничения:

- полный текст `SKILL.md` не должен попадать в catalog;
- явная активация skill должна работать как раньше;
- изменение должно быть видно в тесте provider-а.

Критерий готовности: catalog остаётся полезным, но не раздувает prompt.

### 2. Добавить trace-событие для auto-activation

Цель: сделать выбор skill наблюдаемым.

Задание: если модель вызывает `activate_skill`, запишите в trace имя skill и
способ активации.

Ограничения:

- не писать полный текст `SKILL.md` в trace;
- не дублировать список всех ресурсов;
- сохранить обычное observation для модели.

Критерий готовности: по trace понятно, когда skill был подключён и почему
следующий model call получил новые инструкции.

### 3. Документировать documented scripts внутри skill

Цель: связать инструкции skill с реальными файлами ресурсов.

Задание: добавьте в документацию пример skill с `scripts/`, где markdown
объясняет, какую команду можно запускать через `run_shell`.

Ограничения:

- не добавлять новый механизм исполнения scripts;
- команда должна проходить обычную shell-политику;
- пример должен оставаться локальным для workspace.

Критерий готовности: студент понимает, что skill приносит инструкции и файлы,
но не получает особых привилегий.

## Продвинутый уровень

### 1. Реализовать проверку experimental `allowed-tools`

Цель: исследовать границу между workflow-инструкцией и политикой инструментов.

Задание: если skill указал `allowed-tools`, ограничьте доступные инструменты
после его активации или добавьте диагностику, почему поле пока не enforced.

Ограничения:

- не ломать skills без `allowed-tools`;
- отказ должен быть понятен модели и пользователю;
- тест должен показать выбранную семантику.

Критерий готовности: experimental-поле перестаёт быть неочевидным обещанием.

### 2. Добавить lazy resource preview

Цель: улучшить UX skills без раздувания контекста.

Задание: сделайте так, чтобы при активации skill модель получала короткий список
ресурсов, а содержимое читала только по запросу через `read_file`.

Ограничения:

- не читать все ресурсы заранее;
- не включать бинарные файлы в prompt;
- trace должен показывать факт использования ресурса.

Критерий готовности: skill с большим `references/` остаётся дешёвым для
контекста.

### 3. Спроектировать skill для ревью документации

Цель: применить механизм skills к самой учебной методологии репозитория.

Задание: добавьте пример project-local skill, который помогает проверять
`README.md`, `LABS.md` и `docs/` на соответствие возможностям ветки.

Ограничения:

- skill не должен подменять тесты;
- инструкции должны быть короткими;
- ресурсы должны быть минимальными и учебными.

Критерий готовности: агент с этим skill может провести ревью учебной ветки и
найти упоминание недоступной функции.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

Полезные ручные проверки:

```bash
uv run madharness-mini skills list
uv run madharness-mini skills validate
```

После выполнения задачи проверьте:

- какой текст skill добавил в durable context;
- не попал ли полный `SKILL.md` в trace;
- как модель узнаёт о доступных skills;
- сохраняются ли общие правила workspace-политики.
