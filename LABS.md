# Лабораторные работы: 06-subagents

Эта ветка тренирует оркестрацию ролей. Главная идея: субагент — это не
магический ассистент, а дочерний агентский цикл с собственным prompt, списком
разрешённых tools, лимитами и отдельной трассой.

Задания ниже не пытаются предсказать, решит ли модель делегировать работу в
режиме `auto`. Они развивают то, чем управляет сам харнесс: описание ролей,
контракт `delegate_task`, стартовый контекст субагента, безопасные профили,
формат результата и наблюдаемость дерева делегаций.

## Базовый уровень

### 1. Описать роли для выбора исполнителя

Контекст проблемы: сейчас ведущий агент выбирает субагента по имени, профилю и
короткому описанию. Этого достаточно для демонстрации, но мало для устойчивой
оркестрации: модель видит список ролей, однако не получает явных правил, когда
делегировать исследование, планирование, реализацию или проверку.

Задание: расширьте описание субагента карточкой маршрутизации. Добавьте в
frontmatter необязательные поля вроде `when_to_use`, `handoff_requirements` и
`expected_result`. Эти поля должны попадать в описание `delegate_task`, потому
что именно схему инструмента видит ведущая модель перед выбором роли. Команды
`subagents show` и `subagents validate` тоже должны показывать эти поля и
диагностировать слишком длинные значения.

Ограничения:

- не передавайте ведущему агенту полный prompt субагента;
- не добавляйте runtime-зависимости для разбора markdown;
- старые файлы ролей без новых полей должны продолжить работать;
- встроенные роли нужно описать по-разному, а не повторить одну формулу четырежды;
- диагностика длины должна быть предупреждением, если роль всё ещё пригодна.

Критерий готовности: по схеме `delegate_task` видно, какую роль выбрать для
исследования, плана, реализации и ревью, а старые проектные роли не ломаются.

Проверка: создайте проектную роль с карточкой маршрутизации, проверьте вывод
`subagents show`, предупреждение `subagents validate` для слишком длинного поля
и текст схемы `delegate_task`, который получает модель.

### 2. Собрать структурированный пакет передачи

Контекст проблемы: сейчас `delegate_task` передаёт субагенту строку `task` и
необязательный свободный `context`. При такой форме ведущий агент легко отдаёт
исполнителю расплывчатое поручение: без причины делегации, ограничений,
ожидаемого результата и признаков готовности.

Задание: расширьте аргументы `delegate_task` полями `reason`, `constraints`,
`expected_output` и `definition_of_done`. При запуске субагента эти данные
должны попадать в начало дочернего контекста отдельными секциями, а не
склеиваться с задачей одним абзацем. В родительской трассе запишите краткую
сводку пакета передачи: какие поля были заполнены и какой получился размер.

Ограничения:

- старый формат с `task` и `context` должен остаться рабочим;
- слишком длинные поля нужно обрезать с явной пометкой;
- дочерний контекст не должен получать полную историю ведущего агента;
- все новые поля должны быть обычными JSON-совместимыми значениями;
- пустые поля не должны создавать шумные пустые секции.

Критерий готовности: субагент получает самодостаточную задачу с ограничениями и
критерием завершения, а ведущий агент может по трассе понять, какой пакет был
передан.

Проверка: смоделируйте делегацию `researcher` и `implementer`, проверьте
дочерние сообщения, событие трассы и совместимость со старым вызовом
`delegate_task` только с `task` и `context`.

## Основной уровень

### 3. Передавать субагенту доказательства из проекта

Контекст проблемы: ведущий агент часто уже нашёл важные файлы до делегации.
Если передать субагенту только общую задачу, дочерний цикл заново тратит ходы
на поиск той же информации. Если вручную скопировать слишком много текста в
`context`, легко раздуть запрос и потерять границы источников.

Задание: добавьте в `delegate_task` поле `evidence`: список элементов с путём
файла, необязательным диапазоном строк и коротким пояснением, зачем этот
фрагмент нужен роли. Харнесс должен безопасно прочитать эти фрагменты перед
стартом субагента, добавить их в дочерний контекст отдельным блоком и записать
в трассу, какие источники были переданы.

Ограничения:

- пути проходят через общую политику рабочей области;
- защищённые файлы и выход за пределы проекта запрещены;
- большие фрагменты обрезаются с пометкой;
- бинарные файлы не читаются как текст;
- ошибка одного фрагмента не должна скрывать остальные элементы пакета;
- субагент не получает прав, которых нет в его списке разрешённых tools.

Критерий готовности: субагент стартует с проверяемыми выдержками из проекта, а
не с безымянным пересказом ведущего агента.

Проверка: передайте субагенту два разрешённых фрагмента, один защищённый путь и
один слишком большой диапазон. Проверьте дочерний контекст, ответы об ошибках,
события трассы и то, что субагент может продолжить работу с доступными
доказательствами.

### 4. Вернуть результат субагента как рабочий отчёт

Контекст проблемы: после делегации ведущий агент получает финальный ответ
субагента, путь к дочерней трассе и короткую строку со счётчиками. Этого мало
для уверенного следующего шага: непонятно, какие инструменты реально вызывались,
какие файлы были прочитаны, какие файлы были изменены и чем закончилась
дочерняя работа.

Задание: расширьте сводку дочерней трассы. В ответе `delegate_task` и событии
`subagent_finished` должны быть имя роли, итоговый статус, число обращений к
модели, число вызовов инструментов, список использованных инструментов, список
прочитанных файлов, список изменённых файлов и причина остановки, если это
ошибка, превышение лимита ходов или вопрос через `ask_user`.

Ограничения:

- полный дочерний JSONL-файл не должен попадать в ответ модели;
- большие ответы инструментов нужно обрезать;
- путь к дочерней трассе должен сохраниться;
- список изменений не должен учитывать служебные каталоги `.git`,
  `.madharness-mini`, `__pycache__` и `.venv`;
- повреждённая строка в дочерней трассе не должна валить родительский запуск.

Критерий готовности: ведущий агент получает рабочий отчёт, по которому можно
решить, продолжать самому, делегировать следующую роль или остановиться.

Проверка: дайте тестовому субагенту сценарий, где он читает файл, создаёт
markdown-файл, меняет существующий файл и затем завершает работу. Проверьте
ответ `delegate_task`, событие `subagent_finished`, список прочитанных файлов и
список изменённых файлов.

## Продвинутый уровень

### 5. Сделать профиль субагента наблюдаемым

Контекст проблемы: `profile` в аргументах `delegate_task` уже может безопасно
понизить writable-субагента до `read-only`, а попытка повысить read-only роль до
`writable` должна завершаться ошибкой. Но для учебного харнесса этого мало:
человек должен видеть, какой профиль был запрошен, какие tools остались
доступны и какие полномочия были сняты.

Задание: сделайте итоговый профиль и итоговый набор tools явной частью делегации.
В `subagent_started`, дочернем `session_start` или отдельном событии трассы
записывайте исходный профиль роли, запрошенный профиль, итоговый профиль,
итоговый список tools и список tools, снятых при понижении профиля. В observation
`delegate_task` верните краткую сводку профиля. Ошибка попытки повышения должна
быть обычным ответом инструмента с именем роли, исходным профилем и запрошенным
профилем.

Ограничения:

- `profile` не должен выдавать новые tools сверх frontmatter роли;
- понижение до `read-only` снимает `apply_patch`, `write_file` и `run_shell`;
- read-only роль нельзя сделать writable через аргументы `delegate_task`;
- субагент по-прежнему не может получить `delegate_task`;
- не меняйте формат frontmatter ради одного запуска.

Критерий готовности: по трассе и ответу инструмента понятно, какие полномочия
получил субагент и почему попытка повышения была отклонена.

Проверка: проверьте writable-субагента без переопределения профиля,
writable-субагента с запросом `read-only`, read-only роль с запросом `writable`
и роль, у которой в tools нет пишущих инструментов. Убедитесь, что дочерний
prompt и схемы tools соответствуют итоговому профилю.

### 6. Показать дерево делегаций вместе с передачей контекста

Контекст проблемы: у каждого субагента уже есть дочерняя JSONL-трасса, а
родительская трасса хранит события `subagent_started` и `subagent_finished`.
Команда `trace` пока показывает только короткую сводку по именам ролей, поэтому
учащемуся трудно восстановить, кто что получил, что сделал и где смотреть
подробности.

Задание: расширьте команду `trace`, чтобы по одному идентификатору родительской
трассы было видно дерево делегаций. В сводке покажите каждую роль, путь к её
дочерней трассе, итоговый статус, число ходов, использованные tools,
изменённые файлы, короткий финальный ответ и сводку пакета передачи: какие
требования, ограничения и доказательства получил субагент на старте. Если
субагент остановился через `ask_user`, в дереве должен быть виден вопрос.

Ограничения:

- не печатайте полный prompt, полные ответы инструментов и большие JSON-объекты;
- не читайте файлы за пределами каталога трасс;
- вывод должен оставаться коротким даже при нескольких субагентах;
- точный идентификатор родительской трассы имеет приоритет над похожими
  дочерними именами;
- отсутствующая или повреждённая дочерняя трасса должна давать предупреждение,
  а не ломать просмотр родительской трассы.

Критерий готовности: по одному trace id понятно, какие роли участвовали в
задаче, что им передали и где открыть подробный дочерний журнал.

Проверка: подготовьте родительскую трассу с двумя дочерними запусками: один
успешный, второй с вопросом `ask_user`. Проверьте вывод
`madharness-mini trace`, случай отсутствующей дочерней трассы и случай битой
строки JSONL в дочерней трассе.

## Общая проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

Полезные ручные проверки:

```bash
uv run madharness-mini subagents list
uv run madharness-mini subagents show planner
uv run madharness-mini subagents validate
uv run madharness-mini run --orchestrate-required "Разбей задачу между ролями"
uv run madharness-mini trace <trace-id>
```

После выполнения задачи проверьте:

- какой набор tools получил ведущий агент;
- какой итоговый профиль и набор tools получил субагент;
- какой пакет передачи попал в дочерний контекст;
- где лежит дочерний trace;
- не появилась ли рекурсивная делегация.
