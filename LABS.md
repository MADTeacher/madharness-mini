# Лабораторные работы: 02-AGENTS-md

Эта ветка тренирует работу с проектными инструкциями `AGENTS.md`. Главный
вопрос главы: как harness добавляет к задаче пользователя глобальные и локальные
правила проекта, показывает их происхождение и сохраняет понятные границы
безопасности.

## Базовый уровень

### 1. Добавить глобальный `AGENTS.md` и trace источников

Цель: расширить модель проектных инструкций от «только текущий workspace» до
связки «глобальные правила пользователя + локальные правила проекта».

Задание: сейчас `load_project_instructions()` ищет только `AGENTS.md` от
`Config.root` до `Config.cwd`. Добавьте собственный глобальный уровень
`madharness-mini`: harness должен уметь читать пользовательский `AGENTS.md` из
каталога, заданного `MADHARNESS_MINI_HOME`, а если переменная не задана, из
`~/.madharness-mini/AGENTS.md`.

Глобальные инструкции идут первыми и имеют самый низкий приоритет: локальный
корневой `AGENTS.md` и вложенные `AGENTS.md` уточняют их.

Ограничения:

- не читать глобальный файл за пределами явно выбранного пути;
- не падать, если глобального файла нет;
- не писать полный текст инструкций в отдельное trace-событие;
- сохранить общий лимит размера инструкций.

Критерий готовности: trace показывает список источников инструкций: global,
root, nested, в порядке применения. Модель получает system prompt с глобальными
правилами перед локальными.

Проверка: в тесте создать временный `MADHARNESS_MINI_HOME/AGENTS.md`, корневой
`AGENTS.md` и вложенный `src/AGENTS.md`; порядок должен быть: global, root, src.

### 2. Добавить проверку применимых `AGENTS.md` для целевых файлов

Цель: показать проблему, когда harness запущен из корня, но агент редактирует
файл глубоко внутри проекта, где есть свои локальные инструкции.

Задание: сейчас инструкции выбираются по `Config.cwd`, а не по пути файла, с
которым работает инструмент. Добавьте диагностику для файловых инструментов:
когда агент читает или меняет файл в подпапке, где есть более вложенный
`AGENTS.md`, который не попал в стартовый prompt, observation должен явно
сообщить модели, что для этого пути есть дополнительные инструкции.

Минимальный вариант: добавить в `read_file`, `write_file` и `apply_patch` поле
вроде `applicable_instructions`, где перечислены относительные пути найденных
`AGENTS.md`.

Ограничения:

- не добавлять новый context layer;
- не менять сам prompt во время tool call;
- не читать полный текст дополнительных инструкций автоматически;
- не ломать текущие observations.

Критерий готовности: если harness запущен из корня, а агент читает
`services/payments/app.py`, observation сообщает о `services/AGENTS.md` и
`services/payments/AGENTS.md`, если они существуют.

Проверка: создать проект с `AGENTS.md` в корне и во вложенной папке, вызвать
файловый tool на файл внутри вложенной папки и убедиться, что модель получает
подсказку учесть локальные правила.

## Основной уровень

### 1. Сделать диагностику загрузки инструкций машинно-читаемой

Цель: понять, почему загрузчик инструкций не должен возвращать только готовую
строку prompt.

Задание: сейчас `load_project_instructions()` возвращает готовый текст.
Расширьте механизм так, чтобы вместе с текстом можно было получить диагностику:
какие файлы найдены, какие использованы, какие пустые пропущены, сколько байт
добавлено, был ли текст обрезан по `PROJECT_DOC_MAX_BYTES`.

Ограничения:

- внешний запрос к модели должен остаться совместимым;
- пустой `AGENTS.md` не считать ошибкой;
- диагностика не должна содержать полный текст инструкций.

Критерий готовности: тест может проверить не только итоговую строку, но и
причины: «пустой файл пропущен», «вложенный файл использован», «последний файл
обрезан».

Проверка: создать цепочку из трёх `AGENTS.md`, один пустой и один слишком
большой; диагностика должна честно описывать результат.

### 2. Добавить CLI-команду для просмотра цепочки `AGENTS.md`

Цель: дать человеку понятный способ проверить, какие инструкции будут добавлены
в prompt, не вызывая модель и не разбирая trace руками.

Задание: добавьте CLI-команду, например `madharness-mini instructions`, которая
без обращения к LLM печатает цепочку инструкций, найденных для текущего запуска.
В выводе должны быть видны: уровень источника (`global`, `workspace`, `nested`),
путь, размер файла, статус (`used`, `empty`, `truncated`, `missing`) и итоговый
размер текста, который попадёт в system prompt.

Команда должна отвечать на простой вопрос студента: «почему модель получила
именно эти правила и в таком порядке?».

Ограничения:

- команда не должна обращаться к модели;
- не печатать полный текст `AGENTS.md` по умолчанию;
- не читать файлы вне глобального и workspace-путей;
- вывод должен быть стабильным, чтобы его можно было покрыть CLI-тестом.

Критерий готовности: `madharness-mini instructions` показывает таблицу или
короткий список источников инструкций в том же порядке, в котором они будут
добавлены в prompt.

Проверка: временный проект с global/root/nested инструкциями; CLI показывает
порядок, пустые пропуски и факт обрезания без запуска `ask`.

## Продвинутый уровень

### 1. Усилить правила override и приоритета `AGENTS.md`

Цель: приблизить поведение к модели, где общие правила задают фон, а вложенные и
override-правила уточняют или переопределяют их для своей области.

Задание: сейчас `AGENTS.override.md` явно игнорируется тестом. Спроектируйте и
реализуйте понятную семантику переопределения. Вариант: если рядом с обычным
`AGENTS.md` есть `AGENTS.override.md`, он добавляется после обычного файла той
же директории и помечается как override-фрагмент, который имеет больший
приоритет для этой директории.

Если обычного файла нет, override всё равно может примениться, но
trace/diagnostics должны показывать это явно. Более вложенные override-файлы
идут позже и переопределяют более общие правила в пределах своей области.

Ограничения:

- не пытаться удалять уже прочитанный текст старых правил;
- переопределение задаётся порядком и явными заголовками;
- диагностика должна отличать normal и override sources.

Критерий готовности: итоговый prompt и diagnostics показывают, какие правила
базовые, какие override, и почему более поздний override-фрагмент переопределяет
правило для своей области.

Проверка: root задаёт «используй pytest», вложенный `AGENTS.override.md` задаёт
«для этой папки используй unittest»; override идёт позже и явно помечен.

### 2. Добавить ревью-проверку инструкций

Цель: сделать `AGENTS.md` не просто текстом, а проверяемым учебным контрактом.

Задание: добавьте лёгкий validator для инструкций. Он должен находить очевидно
плохие вещи в `AGENTS.md`: секретоподобные строки, слишком длинный файл, пустой
файл, ссылки на механизмы будущих веток вроде skills/MCP/hooks, если их ещё нет
в этой ветке, и конфликтующие локальные правила без override-пометки.

Это может быть отдельная CLI-команда или часть `instructions` preview.

Ограничения:

- validator не должен блокировать запуск `ask/run`;
- это диагностика для человека, а не новая политика выполнения;
- правила должны быть простыми, без внешних зависимостей;
- не вводить сложный парсинг Markdown.

Критерий готовности: пользователь получает warning-и, которые помогают почистить
`AGENTS.md`, но harness продолжает работать.

Проверка: создать `AGENTS.md` с пустыми секциями, псевдо-секретом и упоминанием
MCP; validator показывает предупреждения, а обычная загрузка инструкций остаётся
рабочей.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

После выполнения задачи проверьте:

- какие инструкции реально увидела модель;
- откуда пришли глобальные, корневые и вложенные правила;
- не попали ли секреты или лишние файлы в trace;
- какая часть поведения покрыта тестом.
