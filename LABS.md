# Лабораторные работы: 03-Context-Layer

Эта ветка тренирует контекстную инженерию внутри harness: что именно модель
видит перед вызовом, как хранится история tool calls и что происходит, когда
запрос становится слишком большим.

## Базовый уровень

### 1. Добавить в `context_report` число закреплённых фрагментов

Цель: понять, чем закреплённый контекст отличается от обычной истории.

Задание: расширьте отчёт контекста полем с количеством pinned/fragments, которые
не удаляются при обрезке истории.

Ограничения:

- не добавлять полный текст prompt в trace;
- сохранить существующие поля отчёта;
- покрыть изменение тестом `ContextManager`.

Критерий готовности: trace показывает размер и состав контекста без утечки
содержимого инструкций.

### 2. Проверить, что `AGENTS.md` переживает обрезку истории

Цель: увидеть роль закреплённых фрагментов на практике.

Задание: добавьте тест, где история переполняет бюджет, но проектные инструкции
остаются в итоговых `messages`.

Ограничения:

- не использовать реальный tokenizer;
- не завязываться на конкретную модель;
- тест должен быть устойчив к небольшим изменениям текста prompt.

Критерий готовности: старые tool observations удаляются или укорачиваются, а
проектные правила остаются видимыми модели.

### 3. Сделать ошибку переполнения контекста понятнее

Цель: разобраться, где harness останавливает слишком большой запрос до API.

Задание: улучшите сообщение `context budget exceeded after truncation`, чтобы
оно подсказывало пользователю, какие настройки можно изменить.

Ограничения:

- не скрывать сам факт ошибки;
- не отправлять слишком большой запрос модели;
- добавить тест на сообщение об ошибке.

Критерий готовности: пользователь понимает, что менять: задачу, историю,
`context_max_tokens` или набор tools.

## Основной уровень

### 1. Настроить стратегию обрезки tool output

Цель: понять, почему длинные observations опасны для следующих model calls.

Задание: измените или расширьте тесты так, чтобы длинный вывод инструмента
обрезался раньше, чем удаляется вся история.

Ограничения:

- пометка `context clipped` должна оставаться явной;
- assistant/tool пара не должна разрываться;
- поведение должно быть видно в `context_report`.

Критерий готовности: отчёт показывает, какие tool messages были обрезаны.

### 2. Добавить учебный `ContextProvider`

Цель: потренировать расширение контекста без изменения основного loop.

Задание: создайте тестовый provider, который добавляет короткий transient
fragment, и проверьте, что он попадает в `messages`.

Ограничения:

- provider должен использовать существующий protocol;
- не добавлять новый runtime-механизм;
- transient fragment не должен вести себя как project instructions.

Критерий готовности: тест показывает, что контекст можно расширять отдельным
объектом, не импортируя `loop.py`.

### 3. Разделить отчёты `ask` и `run`

Цель: увидеть, что tools тоже занимают бюджет.

Задание: сделайте документацию или trace-сводку яснее: в `ask` бюджет считается
только по messages, а в `run` ещё и по tool schemas.

Ограничения:

- не менять API `ModelClient`;
- не дублировать большие schemas в trace;
- сохранить компактность команды `trace`.

Критерий готовности: студент по trace понимает, почему `run` может быть тяжелее
`ask` при одинаковой задаче.

## Продвинутый уровень

### 1. Спроектировать priority-based eviction

Цель: попробовать более тонкую стратегию управления контекстом.

Задание: добавьте приоритеты не только фрагментам, но и элементам истории, чтобы
важные observations можно было сохранять дольше обычных.

Ограничения:

- старые тесты бюджета должны остаться осмысленными;
- приоритет не должен позволять превысить жёсткий лимит;
- в отчёте должно быть видно, что было удалено.

Критерий готовности: поведение объяснимо на маленьком тестовом сценарии.

### 2. Добавить summary для удалённой истории

Цель: исследовать компромисс между полнотой и размером контекста.

Задание: вместо полного удаления старой истории попробуйте оставлять короткое
служебное summary о том, что было удалено.

Ограничения:

- summary не должно придумывать факты;
- оно должно быть явно помечено как служебное;
- если summary тоже не помещается, жёсткий бюджет всё равно важнее.

Критерий готовности: тест показывает, что модель получает краткую память о
старых действиях без тяжёлых tool outputs.

### 3. Добавить учебный отчёт о составе запроса

Цель: сделать context engineering наблюдаемой для учащегося.

Задание: расширьте команду `trace`, чтобы она показывала последнюю сводку
контекста: messages estimate, tools estimate и факт обрезки.

Ограничения:

- не выводить содержимое prompt;
- оставить вывод коротким;
- покрыть формат тестом.

Критерий готовности: после любого запуска студент может понять, из чего
состоял последний запрос к модели.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

После выполнения задачи проверьте:

- какие фрагменты были закреплены;
- какие observations попали в историю;
- что было обрезано или удалено;
- почему итоговый запрос поместился или не поместился в бюджет.
