# Лабораторные работы: 05-mcp

Эта ветка тренирует подключение внешних tools через MCP. Главный учебный фокус:
внешний процесс должен выглядеть для модели как обычный tool, но harness обязан
явно управлять запуском, именами, ошибками, окружением и завершением процесса.

## Базовый уровень

### 1. Проверить чтение выключенного MCP-сервера

Цель: понять, почему MCP должен включаться явно.

Задание: добавьте тест, где `.madharness-mini/mcp.json` содержит сервер с
`enabled=false`, и проверьте, что он не запускается.

Ограничения:

- disabled server не должен создавать процесс;
- ошибка в disabled server не должна ломать запуск;
- поведение должно быть видно в тесте конфига или provider-а.

Критерий готовности: harness игнорирует выключенный сервер и оставляет базовые
инструменты доступными.

### 2. Проверить префиксы имён MCP tools

Цель: увидеть, как harness избегает конфликтов между внешними серверами.

Задание: добавьте тест, где два MCP-сервера возвращают tool с одинаковым именем,
а registry получает разные имена с префиксами.

Ограничения:

- формат `mcp__server__tool` должен остаться стабильным;
- имя сервера должно проходить существующую валидацию;
- тест не должен запускать реальный внешний MCP-сервер.

Критерий готовности: модель видит разные tool names и не теряет источник tool.

### 3. Документировать безопасное окружение MCP

Цель: понять, какие секреты нельзя передавать внешнему процессу автоматически.

Задание: уточните документацию и тест, что MCP-сервер не наследует
`MADHARNESS_MINI_API_KEY` и другие настройки модели.

Ограничения:

- явный `env` из `mcp.json` должен продолжить работать;
- не раскрывать секреты в trace;
- не менять основной `.madharness-mini/config.json`.

Критерий готовности: студент может объяснить, почему MCP-процесс получает
только безопасный минимум окружения.

## Основной уровень

### 1. Улучшить observation для MCP errors

Цель: разобраться, как MCP `isError` превращается в `fail()`.

Задание: сделайте observation для ошибочного `tools/call` более полезным:
summary должен включать сервер, tool и короткую причину.

Ограничения:

- не ломать формат успешных observations;
- большие MCP content blocks должны проходить через clipping;
- тест должен покрыть `isError=true`.

Критерий готовности: модель получает понятную ошибку и может выбрать другой
шаг, а не видит сырой JSON-RPC шум.

### 2. Закрывать MCP-процессы при ошибке model loop

Цель: увидеть lifecycle внешних процессов в agent loop.

Задание: добавьте тест, который доказывает, что `ToolRegistry.close()` или
эквивалентный cleanup вызывается, даже если model call падает.

Ограничения:

- не оставлять дочерние процессы после теста;
- не скрывать исходную ошибку model call;
- cleanup должен быть идемпотентным.

Критерий готовности: внешний процесс закрывается при успехе, ошибке и max turns.

### 3. Добавить trace для lifecycle MCP

Цель: сделать интеграцию внешних tools наблюдаемой.

Задание: запишите события старта сервера, успешного `tools/list` и ошибки
MCP-сервера в trace.

Ограничения:

- не писать секреты из env;
- не дублировать полные schemas всех tools, если это раздувает trace;
- ошибки должны оставаться понятными пользователю.

Критерий готовности: по trace видно, какой MCP-сервер стартовал и какие tools
он добавил.

## Продвинутый уровень

### 1. Ввести allowlist команд MCP-серверов

Цель: обсудить границу доверия к внешним tools.

Задание: добавьте настройку или политику, которая разрешает запускать только
ожидаемые команды MCP-серверов.

Ограничения:

- не использовать shell-строки;
- отказ должен происходить до запуска процесса;
- docs должны объяснять, чем это отличается от `allow_shell`.

Критерий готовности: случайный `command` в `mcp.json` блокируется до старта.

### 2. Добавить поддержку MCP text/resource content

Цель: изучить нормализацию разных типов MCP content.

Задание: расширьте `results.py`, чтобы он аккуратно обрабатывал несколько типов
content blocks, которые реально возвращает MCP-сервер.

Ограничения:

- неподдерживаемые блоки должны превращаться в понятную диагностику;
- большие строки должны обрезаться;
- observation должен оставаться JSON-совместимым.

Критерий готовности: тесты покрывают несколько content blocks в одном ответе.

### 3. Спроектировать MCP smoke server для тестов

Цель: сделать учебные тесты MCP наглядными без внешних зависимостей.

Задание: добавьте маленький fake stdio MCP server в tests fixtures или helper,
который умеет `initialize`, `tools/list` и один `tools/call`.

Ограничения:

- server должен быть на стандартной библиотеке Python;
- тесты не должны ходить в сеть;
- процесс должен гарантированно завершаться.

Критерий готовности: MCP-интеграция проверяется end-to-end без реального
Playwright, npx или внешнего пакета.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

После выполнения задачи проверьте:

- какие процессы были запущены;
- какие tool names получила модель;
- что попало в trace;
- какие секреты не ушли во внешний MCP-процесс.
