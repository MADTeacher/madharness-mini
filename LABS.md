# Лабораторные работы: 02-AGENTS-md

Эта ветка тренирует работу с проектными инструкциями и мультимодальным вводом.
Главный вопрос главы: как harness добавляет к задаче пользователя локальные
правила проекта и при этом сохраняет понятные границы безопасности.

## Базовый уровень

### 1. Показать, какие `AGENTS.md` были найдены

Цель: разобраться в порядке поиска проектных инструкций.

Задание: добавьте в trace событие с относительными путями `AGENTS.md`, которые
попали в system prompt.

Ограничения:

- не записывать полный текст инструкций в новое событие;
- сохранить текущий порядок от корня к вложенной папке;
- добавить тест на корневой и вложенный `AGENTS.md`.

Критерий готовности: trace помогает понять, какие правила увидела модель, не
дублируя весь prompt.

### 2. Улучшить ошибку пустого `AGENTS.md`

Цель: понять, как loader отличает полезные инструкции от шума.

Задание: сделайте так, чтобы пустые или пробельные `AGENTS.md` не попадали в
system prompt, а поведение было явно покрыто тестом.

Ограничения:

- не считать пустой файл ошибкой;
- не менять лимит `PROJECT_DOC_MAX_BYTES`;
- не ломать вложенные инструкции.

Критерий готовности: пустой файл игнорируется, а непустой файл ниже по дереву
всё равно загружается.

### 3. Проверить выключенный vision input

Цель: увидеть, как инструмент сообщает модели о недоступной возможности.

Задание: добавьте или уточните тест, где `read_image` вызывается при
`supports_image_input=false`.

Ограничения:

- инструмент не должен читать и прикреплять изображение в этом режиме;
- observation должен быть `ok=false`;
- сообщение должно объяснять, какое поле конфига включить.

Критерий готовности: модель получает понятный отказ вместо скрытой ошибки.

## Основной уровень

### 1. Добавить диагностику размера проектных инструкций

Цель: понять, почему у `AGENTS.md` есть общий лимит.

Задание: верните из loader не только текст, но и диагностические данные:
сколько файлов прочитано и был ли текст обрезан по лимиту.

Ограничения:

- не менять публичный формат prompt без необходимости;
- не добавлять tokenizer;
- тест должен покрыть переполнение лимита.

Критерий готовности: trace или тестовая диагностика показывает факт обрезки
инструкций.

### 2. Сделать `read_image` строже к форматам

Цель: разобраться, где проходит граница между локальным файлом и модельным
вводом.

Задание: добавьте тесты для неподдерживаемого расширения, слишком большого
файла и анимированного GIF, если соответствующая проверка уже есть в коде.

Ограничения:

- не передавать base64 в trace;
- не читать файл за пределами workspace;
- не добавлять внешние библиотеки для проверки изображений.

Критерий готовности: `read_image` даёт разные понятные отказы для разных причин.

### 3. Добавить пример `AGENTS.md` в документацию

Цель: сделать ветку удобной для студента, который подключает свой маленький
проект.

Задание: обновите docs так, чтобы там был короткий пример хорошего `AGENTS.md`
для Python CLI-проекта.

Ограничения:

- пример должен быть коротким;
- не добавлять правила про будущие skills, MCP или hooks;
- связать пример с командами проверки этой ветки.

Критерий готовности: новый пользователь может скопировать структуру правил и
понять, зачем она нужна.

## Продвинутый уровень

### 1. Разделить встроенный prompt и проектные правила в trace

Цель: подготовиться к будущему слою контекста.

Задание: не меняя API модели, сделайте внутреннее представление стартовых
сообщений более явным: отдельно храните встроенный system prompt и проектные
инструкции перед финальной склейкой.

Ограничения:

- Chat Completions запрос должен остаться совместимым;
- существующие тесты `ask` и `run` должны пройти;
- не вводить новый пакет контекста в этой ветке.

Критерий готовности: код легче читать, но архитектурный скачок к context layer
остаётся задачей следующей ветки.

### 2. Добавить безопасный предпросмотр изображения

Цель: улучшить наблюдаемость `read_image`, не раздувая trace.

Задание: добавьте в observation метаданные изображения: формат, размер файла и
относительный путь.

Ограничения:

- не писать base64 в observation;
- не раскрывать абсолютные пути, если они не нужны пользователю;
- сохранить совместимость follow-up messages для модели.

Критерий готовности: trace показывает, какое изображение было прикреплено, но
не содержит тяжёлого бинарного payload.

### 3. Сформировать правила конфликта вложенных `AGENTS.md`

Цель: продумать методологию локальных инструкций.

Задание: добавьте документацию и тестовый пример, где корневые правила задают
общую политику, а вложенные уточняют стиль конкретной папки.

Ограничения:

- вложенные правила не должны стирать корневые;
- порядок склейки должен оставаться стабильным;
- пример должен быть привязан к реальному поведению loader.

Критерий готовности: студент может объяснить, почему порядок инструкций важен
для harness-инженерии.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

После выполнения задачи проверьте:

- какие инструкции реально увидела модель;
- не попали ли секреты или лишние файлы в trace;
- как observation объясняет отказ инструмента;
- какая часть behavior покрыта тестом.
