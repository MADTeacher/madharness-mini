# Лабораторные работы: 01-minimalistic-harness

Эта ветка тренирует базовое понимание агентского harness: CLI, model loop,
инструменты, workspace-политику и trace. В задачах нет оценки времени: важнее
пройти цикл анализа, правки, проверки и объяснения результата.

## Базовый уровень

### 1. Сделать trace-сводку понятнее

Цель: разобраться, как `trace.py` пишет JSONL, как команда `trace` читает
события запуска и какие данные помогают человеку быстро понять сессию.

Задание: сейчас команда `trace` показывает минимум: путь, число событий,
tool calls и результат. Сделайте выводимую сводку понятнее для человека,
добавив:

- длительность сессии;
- режим запуска `ask` или `run`;
- список инструментов, вызванных в ходе сессии;
- последний статус: успех, ошибка модели или превышение `max_turns`;
- короткий вывод по каждому tool call.

Ограничения:

- не добавлять runtime-зависимости;
- не ломать чтение старых JSONL-трасс;
- не выводить в терминал полные большие payload инструментов;
- покрыть новое поведение тестом или маленьким fixture-trace.

Критерий готовности: `madharness-mini trace <id>` показывает компактную
человеческую сводку, по которой понятно, чем закончился запуск и какие tools
участвовали.

Проверка: выполните короткий `ask`, затем `run` с одним инструментом, откройте
оба trace через CLI и сравните, отличается ли режим, статус и список tool calls.

### 2. Улучшить навигацию через `list_files`

Цель: понять, как файловые инструменты помогают агенту ориентироваться в
workspace и как ограничивать результат, не лишая модель полезной информации.

Задание: инструмент `list_files` возвращает до 200 файлов и фильтрует glob
только по имени. Улучшите навигацию по проекту:

- добавьте стабильную сортировку результата;
- добавьте параметр `max_results`;
- сделайте glob по относительному пути, например
  `madharness_mini/tools/*.py`;
- явно сообщайте LLM, если результат был обрезан.

Ограничения:

- не показывать служебные каталоги вроде `.git`, `.madharness-mini`,
  `__pycache__` и `.venv`;
- не выходить за workspace-границы `Policy.safe_path()`;
- сохранить предсказуемый формат observation;
- если проблема навигации решается описанием tool-а, сначала улучшайте
  описание для LLM, а не усложняйте код.

Критерий готовности: агент может предсказуемо получить ограниченный,
отсортированный список файлов и понять, что выдача была обрезана.

Проверка: попросите агента найти файлы инструментов, файлы CLI и markdown-файлы.
Убедитесь, что повторный запуск даёт стабильный порядок и что glob по
относительному пути работает ожидаемо.

## Основной уровень

### 1. Показать изменения workspace после `run`

Цель: увидеть, как agent loop может наблюдать последствия своей работы без
обращения к git.

Задание: после выполнения `run` пользователь видит финальный ответ агента и путь
к trace, но harness не показывает, какие файлы были изменены за время сессии.
Добавьте в `run_agent` простой отчёт об изменениях проекта:

- в начале запуска сохраните снимок файлов в рабочей директории;
- для каждого файла храните относительный путь, размер и время изменения или
  хеш содержимого для небольших файлов;
- в конце запуска сравните состояние workspace с начальным;
- определите добавленные, изменённые и удалённые файлы;
- запишите отдельное событие в trace, например `workspace_changes`;
- покажите краткую сводку пользователю после завершения `run`.

Ограничения:

- не учитывать служебные и игнорируемые каталоги вроде `.git`,
  `.madharness-mini`, `__pycache__`, `.venv`;
- не использовать `git status` как источник истины;
- не читать большие файлы целиком без необходимости;
- сохранить простую структуру данных, понятную студенту.

Критерий готовности: после `run` пользователь видит краткий список добавленных,
изменённых и удалённых файлов, а trace содержит машинно-читаемое событие с теми
же изменениями.

Проверка: попросите агента создать новый файл, изменить существующий файл и
удалить временный файл. Убедитесь, что harness корректно показывает список
изменений без обращения к git.

### 2. Улучшить `search_code` для выбора следующего файла

Цель: сделать поиск по коду полезным не только для факта совпадения, но и для
решения, какой файл читать дальше.

Задание: инструмент `search_code` ищет только точную подстроку и возвращает одну
строку совпадения. Улучшите его:

- добавьте параметр `context` для вывода нескольких строк вокруг совпадения;
- добавьте параметр `case_sensitive`;
- при `case_sensitive=true` запрос `Trace` должен находить только `Trace`;
- при `case_sensitive=false` запрос должен находить также `trace`, `TRACE` и
  другие варианты регистра;
- добавьте ограничение на размер просматриваемого файла;
- сделайте `summary` информативнее: сколько файлов просмотрено, сколько
  совпадений найдено, был ли результат обрезан.

Ограничения:

- не добавлять внешние поисковые зависимости;
- не выходить за workspace;
- не отдавать модели слишком большие фрагменты;
- явно показывать, если файл пропущен из-за размера.

Критерий готовности: observation поиска помогает агенту выбрать следующий файл
для чтения без лишнего полного `read_file`.

Проверка: попросите агента найти использования `safe_path`, `ToolSpec` и
`trace.write`. Оцените, хватает ли результата поиска для следующего шага без
лишнего чтения всего файла.

## Продвинутый уровень

### 1. Разделить решения shell-политики

Цель: сделать политику shell-команд объяснимой и добавить контролируемое
подтверждение для опасных, но иногда допустимых команд.

Задание: сейчас политика shell-команд устроена как список запрещённых
фрагментов: `rm -rf`, `sudo`, `curl`, `wget` и т.д. Разделите решения политики
на три типа:

- команда разрешена сразу;
- команда запрещена без возможности подтверждения;
- команда требует подтверждения пользователя.

Добавьте режим пользовательской эскалации для команд, которые потенциально
опасны, но в конкретном контексте могут быть допустимы. Например, `rm -rf .`,
`rm -rf /`, `rm -rf ..`, удаление защищённых путей и wildcard-удаление должны
отклоняться сразу, а `rm -rf build` или `rm -rf dist` могут запрашивать
подтверждение.

Если агент в ходе `run` вызывает команду, требующую подтверждения, harness
должен остановиться и спросить пользователя, разрешить ли выполнение. У
пользователя должны быть варианты:

- разрешить один раз;
- запретить;
- разрешить и больше не спрашивать при повторении такой же команды в рамках
  текущей сессии.

Ограничения:

- не разрешать команды за пределами workspace;
- не делать подтверждение глобальным между сессиями;
- явно различать отказ политики и отказ пользователя;
- не скрывать исходную команду от пользователя.

Критерий готовности: `run_shell` возвращает LLM понятную причину: команда
выполнена, запрещена политикой, отклонена пользователем или выполнена после
пользовательского разрешения.

Проверка: попросите агента выполнить безопасную команду, затем команду очистки
вроде `rm -rf build`, затем явно опасную команду вроде `rm -rf .`. Убедитесь,
что все три сценария различаются в observation, выводе CLI и trace.

### 2. Добавить сжатие истории в `run_agent`

Цель: понять, почему длинная история model/tool сообщений мешает долгой работе
агента, даже если отдельные tool results уже обрезаются.

Задание: функция `run_agent`, отвечающая за агентский цикл работы, добавляет в
историю все сообщения модели и все результаты инструментов. Добавьте простое
управление размером истории:

- настройку вроде `max_history_chars`;
- сжатие старых результатов инструментов до коротких summaries при превышении
  лимита;
- событие `history_compacted` в trace.

Полная диагностическая информация при этом должна оставаться в JSONL-трассе.

Ограничения:

- не удалять из trace полные события, нужные для диагностики;
- не сжимать последнее сообщение так, чтобы модель потеряла результат только
  что выполненного tool call;
- не добавлять tokenizer или внешнюю библиотеку;
- summary должно честно описывать, что было сжато, а не придумывать новые факты.

Критерий готовности: при длинной работе harness уменьшает историю для следующих
model calls, а trace показывает, когда и какие tool observations были сжаты.

Проверка: дайте агенту задачу, где он много читает файлов или получает длинный
вывод shell-команды. Убедитесь, что он продолжает работу, а trace показывает
событие `history_compacted` и краткое описание сжатия.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

После выполнения задачи кратко ответьте себе:

- какой слой harness пришлось изменить;
- какие данные увидела модель;
- какая граница безопасности могла сломаться;
- что показала trace после проверки;
- стало ли агенту проще выбрать следующий шаг.
