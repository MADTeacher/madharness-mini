# Лабораторные работы: 06-subagents

Эта ветка тренирует оркестрацию ролей. Главная идея: субагент — это не отдельный
магический ассистент, а дочерний agent loop с собственным prompt, allow-list
tools, лимитами и trace.

## Базовый уровень

### 1. Проверить режимы оркестрации

Цель: понять, когда parent agent видит `delegate_task`.

Задание: добавьте тесты для режимов `off`, `requested`, `auto` и `required`.

Ограничения:

- в `off` tool `delegate_task` не должен появляться;
- в `required` parent должен быть ограничен ролью координатора;
- поведение CLI-флагов должно совпадать с конфигом.

Критерий готовности: тесты показывают, какой набор tools получает parent в
каждом режиме.

### 2. Улучшить диагностику `subagents validate`

Цель: разобраться в loader-е markdown-субагентов.

Задание: сделайте ошибку невалидного frontmatter понятнее: путь, поле и причина
должны быть видны в выводе.

Ограничения:

- валидные встроенные роли не должны шуметь;
- project-local ошибка не должна скрывать другие роли;
- тест должен покрыть невалидный файл.

Критерий готовности: студент по выводу понимает, какой md-файл исправить.

### 3. Проверить ограничение встроенного `planner`

Цель: увидеть, как роли получают не только tools, но и дополнительные write
scope-ограничения.

Задание: добавьте тест, где `planner` может менять `.md`, но не может менять
Python-файл.

Ограничения:

- отказ должен быть обычным observation;
- правило должно касаться и `write_file`, и `apply_patch`;
- другие роли не должны случайно наследовать это ограничение.

Критерий готовности: роль planner остаётся планировщиком, а не реализатором.

## Основной уровень

### 1. Сделать summary `delegate_task` полезнее

Цель: понять, как результат дочернего loop возвращается parent agent.

Задание: расширьте observation `delegate_task`: добавьте имя субагента, статус,
число ходов и ссылку на локальный trace.

Ограничения:

- не вставлять полный дочерний trace в observation;
- не терять финальный ответ субагента;
- тест должен покрыть успешную делегацию.

Критерий готовности: parent получает компактный результат, а человек может
открыть подробный trace отдельно.

### 2. Проверить `ask_user` внутри субагента

Цель: увидеть контролируемую остановку запуска вопросом к пользователю.

Задание: добавьте тест, где субагент вызывает `ask_user`, а основной `run`
завершается статусом `needs_user_input`.

Ограничения:

- `ask_user` не должен читать stdin;
- вопрос должен быть коротким и попасть в итоговый вывод;
- parent не должен продолжать tool loop после такого результата.

Критерий готовности: пользователь видит вопрос субагента как нормальный финал
запуска.

### 3. Добавить project-local override роли

Цель: разобраться, как проект может заменить встроенного субагента.

Задание: добавьте тест и документацию для случая, когда project-local md-файл
использует имя встроенной роли и `override: true`.

Ограничения:

- без `override: true` конфликт должен быть диагностирован;
- override не должен менять другие встроенные роли;
- итоговая роль должна показываться в `subagents show`.

Критерий готовности: студент понимает, как безопасно кастомизировать роли под
проект.

## Продвинутый уровень

### 1. Добавить роль `test-writer`

Цель: спроектировать новую роль на реальном workflow.

Задание: добавьте встроенного или project-local субагента, который отвечает за
минимальные unittest после изменений кода.

Ограничения:

- tools должны быть достаточными, но не чрезмерными;
- роль должна уметь задать вопрос через `ask_user`, если требований мало;
- docs должны объяснять, когда её вызывать.

Критерий готовности: parent может делегировать написание тестов отдельной роли.

### 2. Сжать parent context для субагента

Цель: исследовать передачу контекста между parent и child loop.

Задание: добавьте короткое summary parent context вместо передачи лишней
истории, если задача делегации уже содержит достаточно деталей.

Ограничения:

- не терять исходную задачу делегации;
- не передавать полный trace;
- поведение должно быть видно в тесте или trace.

Критерий готовности: дочерний контекст остаётся маленьким, но полезным.

### 3. Добавить trace-сводку по дереву делегаций

Цель: сделать оркестрацию наблюдаемой для учащегося.

Задание: расширьте команду `trace`, чтобы она показывала дочерние traces,
созданные через `delegate_task`.

Ограничения:

- не читать все дочерние JSONL полностью без необходимости;
- вывод должен оставаться коротким;
- тест должен покрыть хотя бы одну делегацию.

Критерий готовности: по одному trace id понятно, какие роли участвовали в
задаче.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

Полезные ручные проверки:

```bash
uv run madharness-mini subagents list
uv run madharness-mini subagents validate
```

После выполнения задачи проверьте:

- какой набор tools получил parent;
- какой набор tools получил субагент;
- где лежит дочерний trace;
- не появилась ли рекурсивная делегация.
