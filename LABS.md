# Лабораторные работы: 07-hooks

Тема ветки — lifecycle hooks как слой проектной политики вокруг агентского
цикла. Сейчас механизм намеренно простой: `loop.py` и `model_loop.py` явно
вызывают `HookManager.emit()`, `HookManager` синхронно перебирает подходящие
обработчики, hook-команда получает компактный JSON через stdin, а остановить
действие может только событие `before_tool_call`.

Лабораторные работы ниже ведут этот механизм к более промышленной архитектуре:
явный контракт событий, проверка конфигурации до запуска модели, разделение
наблюдающих hooks и блокирующих правил, наблюдаемость, внутренняя шина событий
и отложенная запись событий трассировки без потери строгих проверок
безопасности.

## Лёгкий уровень

### 1. Добавить проверку hooks перед запуском

Контекст проблемы: сейчас ошибки `.madharness-mini/hooks.json` проявляются
только во время `ask` или `run`. Для учебного проекта это терпимо, но в
практическом харнессе проектная политика должна проверяться до обращения к
модели: человек должен увидеть, какие hooks включены, какие события они слушают
и почему конфигурация невалидна.

Задание: добавьте команду `madharness-mini hooks validate`. Она должна читать
`.madharness-mini/hooks.json`, применять те же правила, что и
`HookManager.from_config()`, и печатать короткий отчёт: есть ли файл, сколько
hooks включено, какие события используются, какие условия `match` заданы и
какие записи отключены через `enabled: false`.

Ограничения:

- используйте существующий загрузчик конфигурации, а не второй парсер;
- не запускайте внешние hook-команды;
- не печатайте значения `env`;
- отсутствие `hooks.json` должно быть нормальным состоянием;
- не меняйте поведение `ask` и `run`.

Критерий готовности: пользователь может проверить hook-конфиг без модели и
получить те же ошибки валидации, которые остановили бы обычный запуск.

Проверка: добавьте тесты команды для отсутствующего `hooks.json`, корректного
конфига, отключённого hook, неверного `event`, неверного `cwd` и неверного
`timeout_seconds`. Проверьте ручной запуск команды на маленьком проекте.

### 2. Стабилизировать контракт блокировки инструмента

Контекст проблемы: если hook блокирует `before_tool_call`, модель получает
обычный ответ инструмента с `ok=false` и текстом `blocked by hook: ...`. Этого
мало для проверяемой трассировки: в ответе нет стабильного имени обработчика,
машиночитаемого кода причины и признака, что обработчик инструмента не
запускался.

Задание: расширьте результат блокировки. `HookDecision` или связанный объект
решения должен сохранять `hook_id`, короткий `reason_code` и безопасный текст
причины. Ответ инструмента при блокировке должен содержать `hook_blocked: true`,
`hook_id`, `reason_code` и привычный `summary`. Существующий текст
`blocked by hook: ...` можно сохранить для обратной совместимости.

Ограничения:

- не передавайте модели полный hook-конфиг, `env`, stderr или данные события;
- успешные вызовы инструментов не должны получать новые поля;
- сбой hook-процесса пока остаётся разрешением, как в текущей архитектуре;
- `after_tool_call` должен получать тот же ответ инструмента, который увидит
  модель;
- старые hooks, которые возвращают только `block`, должны продолжить работать.

Критерий готовности: по ответу инструмента и по трассе понятно, какой hook
остановил действие, почему оно было остановлено и запускался ли обработчик
инструмента.

Проверка: расширьте тест блокировки `before_tool_call`: проверьте сообщение,
которое попадёт в следующий запрос модели, событие `tool_observation`, событие
`after_tool_call` и отсутствие вызова `ToolRegistry.call()`.

## Средний уровень

### 3. Разделить наблюдающие и блокирующие hooks

Контекст проблемы: текущий `HookManager` относится ко всем hooks почти одинаково:
подходящий обработчик запускается, его ошибка пишется как `hook_failed`, а
первое решение `ok=false` возвращается наверх. В промышленной системе нужно
явно различать наблюдение и применение ограничения. Наблюдающий hook нужен для
логирования, метрик и трассировки; он не должен случайно остановить агента.
Блокирующий hook, наоборот, должен иметь понятную политику поведения при
собственной ошибке.

Задание: добавьте в конфиг hook поле назначения, например
`mode: "observe" | "enforce"`, и поле поведения при ошибке, например
`on_error: "allow" | "block"`. Значение по умолчанию должно сохранить текущее
поведение. Только hook с `mode: "enforce"` может остановить
`before_tool_call`. Если такой hook с `on_error: "block"` падает, возвращает
невалидный JSON или превышает таймаут, обработчик инструмента не запускается, а
модель получает обычный ответ инструмента с `ok=false`.

Ограничения:

- старые конфиги без новых полей должны работать как раньше;
- `mode: "observe"` никогда не блокирует действие, даже если hook вернул
  `ok=false`;
- `on_error: "block"` имеет силу только для `before_tool_call`;
- аварийная блокировка должна отличаться в трассе от явного `hook_blocked`;
- stderr и большие сообщения об ошибке должны обрезаться.

Критерий готовности: проект может отдельно описать hooks для наблюдаемости и
hooks для применения ограничений, а поведение при падении проверки становится
явной частью конфигурации.

Проверка: сравните четыре сценария: hook с `mode: "observe"` возвращает block,
hook с `mode: "enforce"` возвращает block, hook с `mode: "enforce"` падает при
`on_error: "allow"` и hook с `mode: "enforce"` падает при
`on_error: "block"`. Проверьте ответ инструмента, события трассы и факт, что
обработчик инструмента не запускается только в блокирующих сценариях.

### 4. Сделать цепочку hooks наблюдаемой

Контекст проблемы: JSONL-трасса уже содержит `hook_started`, `hook_finished`,
`hook_blocked` и `hook_failed`, но сводка `madharness-mini trace <id>` не
показывает путь решения. Учащийся не видит, какие hooks были кандидатами,
какие реально сработали, сколько заняли времени, кто разрешил действие и кто
остановил запуск инструмента.

Задание: добавьте в трассу и сводку команды `trace` представление цепочки hooks.
Для каждого hook-события нужно видеть число подходящих обработчиков, порядок
выполнения, итоговое решение, задержку, блокирующий `hook_id`, если он был, и
число пропущенных обработчиков. Команда `trace` должна показывать компактную
строку по hooks без полного набора данных события.

Ограничения:

- не печатайте аргументы инструмента, ответы модели, `env` и полный набор
  данных события;
- вывод `trace` должен оставаться коротким при большом числе событий;
- точный поиск trace id должен остаться приоритетнее поиска по префиксу;
- отсутствие hooks не должно добавлять шум в сводку;
- повреждённая строка JSONL должна давать понятное предупреждение или
  игнорироваться по уже принятому в проекте правилу.

Критерий готовности: по одному `trace <id>` можно восстановить, как цепочка
hooks приняла решение по опасному вызову инструмента.

Проверка: подготовьте искусственную трассу и интеграционный тест с двумя
наблюдающими hooks, одним hook с `mode: "enforce"` и одним падающим hook.
Проверьте порядок, счётчики, задержки, блокировку и компактность вывода.

## Продвинутый уровень

### 5. Перепроектировать hooks на внутреннюю шину событий

Контекст проблемы: сейчас hooks подключены точечно: `loop.py`,
`model_loop.py` и `subagents/runner.py` напрямую вызывают `HookManager.emit()`.
Это прозрачно для учебной ветки, но плохо масштабируется. Как только появляются
новые подписчики — трасса, метрики, проектная политика, внешние hooks,
отложенная запись событий трассировки, события субагентов, — явные вызовы
`HookManager` превращаются в связанный код без общего контракта публикации
событий.

Задание: спроектируйте и внедрите внутреннюю шину событий. Введите объект вроде
`LifecycleEventBus`, который принимает событие один раз и рассылает его
подписчикам: подписчику трассы, подписчику hook-команд и, при необходимости,
другим локальным подписчикам. Блокирующие события должны возвращать
агрегированное решение, а обычные события должны оставаться наблюдательными.
Старый внешний контракт `.madharness-mini/hooks.json` должен сохраниться.

Ограничения:

- не добавляйте зависимости времени выполнения;
- порядок обработки подписчиков должен быть детерминированным;
- `before_tool_call` по-прежнему должен уметь остановить обработчик до запуска;
- `session_start`, `before_model_call`, `after_model_call`, `after_tool_call`,
  `session_end` и `session_error` не должны внезапно менять ход выполнения;
- трасса должна сохранить совместимые имена событий или иметь явную миграцию;
- субагенты должны использовать ту же шину, но с дочерней трассой и
  собственными метаданными.

Критерий готовности: события жизненного цикла публикуются через единый механизм,
а добавление нового подписчика не требует править весь агентский цикл.

Проверка: перенесите существующие тесты hooks на новую шину без потери
поведения. Добавьте тест на порядок подписчиков, блокировку `before_tool_call`,
ошибку наблюдающего подписчика, дочерний запуск субагента и отсутствие
hooks-конфига.

### 6. Развести синхронные проверки и отложенную запись трассировки

Контекст проблемы: текущие hook-команды всегда выполняются синхронно. Это
просто и безопасно для `before_tool_call`, но плохо подходит для логирования и
наблюдаемости: медленный обработчик трассировки задерживает обращение к модели,
хотя не влияет на решение. В промышленной архитектуре блокирующие проверки
остаются на критическом пути, а запись событий наблюдаемости можно
буферизовать, выполнить позже и всё равно отразить в трассе.

Задание: добавьте режим отложенной записи для hooks, которые не могут
блокировать действие и нужны только для наблюдаемости. Например,
`dispatch: "sync" | "deferred"` или отдельное правило для
`mode: "observe"`. Отложенные события должны попадать в ограниченную очередь,
выполняться до `session_end` или при закрытии `ToolRegistry`, а результат их
работы должен записываться в трассу. Если очередь переполнена или отложенная
запись не успела завершиться за лимит, харнесс должен записать диагностическое
событие, но не ломать основной запуск.

Ограничения:

- hooks с `mode: "enforce"` и `before_tool_call` остаются синхронными;
- порядок событий для одного trace должен быть объяснимым и проверяемым;
- очередь должна иметь явный лимит, чтобы не раздувать память;
- отложенный hook не должен получать больше данных, чем синхронный;
- при ошибке отложенного hook запуск не должен превращаться в блокировку;
- тесты не должны зависеть от случайных задержек.

Критерий готовности: медленный hook наблюдаемости больше не задерживает
критический путь модели и инструментов, но его результат остаётся видимым в
трассе как часть логирования и проверки работы харнесса.

Проверка: создайте сценарии с быстрым синхронным hook с `mode: "enforce"`,
медленным отложенным hook с `mode: "observe"`, переполненной очередью, ошибкой
отложенного hook и сессией субагента. Измерьте порядок событий и убедитесь, что
блокирующие проверки по-прежнему выполняются до запуска инструмента.

## Общая проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

Полезные ручные проверки:

```bash
uv run madharness-mini hooks validate
uv run madharness-mini run "Проверь, какие hooks подключены в проекте"
uv run madharness-mini trace <trace-id>
uv run madharness-mini subagents validate
```

После выполнения любой задачи проверьте:

- какой JSON получает hook-команда;
- где событие публикуется в агентском цикле;
- может ли решение hook остановить только разрешённую точку выполнения;
- как ошибка hook отражается в трассе;
- не раскрылись ли секреты, полный системный запрос, `env` или большие ответы
  инструментов;
- сохранилось ли поведение проекта без `.madharness-mini/hooks.json`.
