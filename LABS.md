# Лабораторные работы: 01-minimalistic-harness

Эта ветка тренирует базовое понимание агентского harness: CLI, model loop,
инструменты, workspace-политику и trace. В задачах нет оценки времени: важнее
пройти цикл анализа, правки, проверки и объяснения результата.

## Базовый уровень

### 1. Сделать trace-сводку понятнее

Цель: разобраться, как `trace.py` пишет JSONL и как команда `trace` показывает
результат пользователю.

Задание: добавьте в сводку количество вызовов модели и инструментов. Сохраните
формат коротким, чтобы вывод оставался удобным в терминале.

Ограничения:

- не менять JSONL-формат уже записанных событий;
- не добавлять зависимости;
- покрыть поведение тестом.

Критерий готовности: `madharness-mini trace <id>` показывает итог, количество
model calls и количество tool observations.

### 2. Улучшить сообщение после `init`

Цель: увидеть, где CLI связывает пользовательские аргументы с локальным
конфигом проекта.

Задание: сделайте вывод `init` более обучающим: после создания конфига покажите,
какой файл записан и какие поля можно менять вручную.

Ограничения:

- не печатать API-ключ;
- не менять структуру `config.json`;
- сохранить работу `--no-prompt`.

Критерий готовности: тест CLI проверяет новый текст без обращения к модели.

### 3. Добавить тест на защищённые пути

Цель: понять роль `Policy.safe_path()` в защите workspace.

Задание: добавьте тест, который доказывает, что путь из `protected_paths`
нельзя прочитать или записать через файловые инструменты.

Ограничения:

- тест должен работать во временной директории;
- не полагаться на реальный `.env` пользователя;
- проверять именно отказ политики, а не случайную ошибку файловой системы.

Критерий готовности: тест падает без защиты и проходит с текущей политикой.

## Основной уровень

### 1. Сделать ошибку неизвестного инструмента полезной

Цель: разобраться в `ToolRegistry` и формате observation.

Задание: если модель вызывает неизвестный tool, верните observation с
перечислением доступных имён инструментов.

Ограничения:

- не раскрывать внутренние Python-объекты;
- не менять формат успешных observation;
- добавить тест на неизвестный инструмент.

Критерий готовности: модель получает `ok=false`, краткую причину и список
доступных tool names.

### 2. Добавить лимит на `list_files`

Цель: увидеть, как инструменты защищают контекст модели от слишком большого
вывода.

Задание: добавьте параметр `limit` к `list_files` и используйте значение по
умолчанию, если модель его не передала.

Ограничения:

- сохранить текущий порядок обхода файлов;
- не показывать служебные каталоги;
- в observation явно указать, был ли список обрезан.

Критерий готовности: тест создаёт больше файлов, чем лимит, и проверяет пометку
об обрезке.

### 3. Сделать `run_shell` диагностичнее

Цель: понять границу между безопасным запуском команды и полезной диагностикой.

Задание: добавьте в observation `run_shell` код возврата команды и признак
таймаута, если он уже есть в обработчике.

Ограничения:

- не разрешать shell-цепочки, пайпы и редиректы;
- не расширять список опасных команд без отдельного теста;
- не менять максимальную длину stdout/stderr.

Критерий готовности: тест проверяет успешную команду и команду с ненулевым
кодом возврата.

## Продвинутый уровень

### 1. Ввести read-only режим для harness

Цель: спроектировать простую настройку, которая запрещает изменения, но
оставляет агенту анализ проекта.

Задание: добавьте поле конфига, которое отключает `write_file`, `apply_patch`
и потенциально изменяющие shell-команды.

Ограничения:

- оставить `ask` без изменений;
- отказ должен приходить как обычное observation `ok=false`;
- тесты должны покрыть файловую правку и shell-команду.

Критерий готовности: в read-only режиме агент может читать и искать, но не может
изменять workspace.

### 2. Добавить dry-run для `apply_patch`

Цель: глубже понять патчинг как главный инструмент правки кода.

Задание: добавьте режим, в котором `apply_patch` проверяет применимость патча,
но не меняет файлы.

Ограничения:

- dry-run не должен создавать, удалять или менять файлы;
- ошибки разбора патча должны быть такими же, как в обычном режиме;
- тест должен сравнить содержимое файла до и после dry-run.

Критерий готовности: модель может проверить патч перед реальной правкой.

### 3. Сформировать учебный smoke-проект

Цель: потренировать harness не на собственном коде, а на маленьком внешнем
workspace.

Задание: добавьте тестовый fixture-проект и сценарий, который запускает агента
на простой задаче без реального обращения к LLM.

Ограничения:

- model client нужно подменить тестовой заглушкой;
- fixture должен быть маленьким и понятным;
- тест не должен требовать API-ключ.

Критерий готовности: тест показывает полный цикл `run`: model call, tool call,
observation и финальный ответ.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

После выполнения задачи кратко ответьте себе:

- какой слой harness пришлось изменить;
- какие данные увидела модель;
- какая граница безопасности могла сломаться;
- что показала trace после проверки.
