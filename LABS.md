# Лабораторные работы: 03-Context-Layer

Эта ветка тренирует контекстную инженерию внутри harness: что именно модель
видит перед вызовом, как хранится история assistant/tool сообщений, как
оценивается бюджет запроса и что происходит, когда контекст становится слишком
большим.

## Базовый уровень

### 1. Расширить `context_report` составом фрагментов

Цель: понять, чем закреплённые фрагменты отличаются от истории assistant/tool
сообщений и почему их нельзя молча удалить при обрезке контекста.

Архитектурный ориентир: в зрелых harness для агентов контекст должен иметь явное
происхождение. По отчёту должно быть видно не содержимое инструкций, а то, из
каких источников они пришли и в какую часть запроса попали.

Задание: сейчас `ContextManager.report()` уже показывает список фрагментов, но
общую картину приходится собирать по отдельным полям отчёта. Добавьте в
`context_report` короткую агрегированную сводку по фрагментам:

- сколько всего фрагментов собрано;
- сколько из них попадает в `system` и `user`;
- сколько временных provider-фрагментов помечено `transient=True`;
- общий размер фрагментов в символах;
- список источников без полного текста.

Сначала устраните потерю происхождения в стартовом контексте: встроенный system
prompt и проектные инструкции `AGENTS.md` сейчас склеиваются в один фрагмент.
Разделите их на два `ContextFragment` или добавьте другой явный и безопасный
механизм указания источников, чтобы `context_report` различал
`madharness_mini/prompts/system.md` и найденные `AGENTS.md`. Формат итоговых
Chat Completions `messages` при этом должен остаться тем же: несколько system
фрагментов всё ещё рендерятся в одно `role=system` сообщение.

После изменения по отчёту должно быть понятно, какие постоянные инструкции
модель увидела перед историей assistant/tool сообщений.

Ограничения:

- не записывать полный prompt или текст `AGENTS.md` в trace;
- сохранить уже существующий список `fragments` в отчёте;
- не терять source каждого project instruction при сборке `base_context()`;
- не менять формат Chat Completions `messages`;
- покрыть поведение тестом `ContextManager`.

Критерий готовности: `context_report` показывает и подробные элементы, и
компактную сводку: placement counts, число временных provider-фрагментов,
total chars и sources. В отчёте отдельно видны встроенный system prompt и
`AGENTS.md`, но их тексты не попадают в trace.

Проверка: создать context с system-фрагментом, `AGENTS.md`-фрагментом,
user-фрагментом и временным provider-фрагментом; убедиться, что report считает
их отдельно, показывает sources и не содержит полного текста.

### 2. Сделать ошибку переполнения контекста обучающей

Цель: разобраться, где harness останавливает слишком большой запрос до API и
как объяснить пользователю причину остановки.

Архитектурный ориентир: ошибка бюджета должна быть структурированной
диагностикой. Она не раскрывает содержимое запроса, но показывает, какая часть
запроса съела бюджет и какие настройки влияют на ситуацию.

Задание: сейчас при жёстком переполнении появляется ошибка
`context budget exceeded after truncation` с общей оценкой токенов. Сделайте её
полезнее: добавьте в текст ошибки `request_tokens_estimate`,
`messages_tokens_estimate`, `tools_tokens_estimate`, `max_tokens`, число
обрезанных tool messages и число удалённых history entries.

Сообщение должно подсказывать, что можно менять: `context_max_tokens`,
`context_keep_recent_turns`, длину задачи, объём tool output или набор доступных
tools.

Ограничения:

- не отправлять заведомо слишком большой запрос модели;
- не скрывать сам факт ошибки;
- не выводить содержимое prompt или tool output;
- сохранить событие `context_error` в trace.

Критерий готовности: пользователь по CLI-ошибке понимает, какая часть запроса
переполнила бюджет и какие настройки относятся к проблеме.

Проверка: в тесте задать очень маленький `context_max_tokens` и большой tool
schema; убедиться, что ошибка содержит estimates и подсказки, а trace содержит
безопасный `context_report`.

## Основной уровень

### 1. Добавить CLI-команду для предпросмотра контекста

Цель: дать способ увидеть оценку будущего запроса без обращения к модели и без
ручного чтения JSONL trace.

Архитектурный ориентир: это предварительная проверка запроса. Пользователь
заранее видит стоимость `ask` и `run`, а harness не смешивает диагностику
контекста с обращением к модели или выполнением инструментов.

Задание: добавьте CLI-команду, например `madharness-mini context "задача"`,
которая строит `ContextManager` так же, как `ask` или `run`, но не вызывает LLM.
Команда должна показывать estimates для messages, tools и всего request,
`context_max_tokens`, список источников фрагментов и факт `over_budget`.

Для режима `run` команда должна учитывать schemas доступных tools, потому что
они тоже занимают контекст. Это можно сделать флагом вроде `--mode ask|run`.

Ограничения:

- команда не должна обращаться к модели;
- не печатать полный system prompt или `AGENTS.md`;
- не выполнять tools;
- вывод должен быть стабильным для CLI-теста.

Критерий готовности: можно сравнить `ask` и `run` для одной задачи и увидеть,
почему `run` дороже по бюджету из-за tool schemas.

Проверка: запустить preview в тестовом проекте с `AGENTS.md`; проверить, что
`--mode ask` показывает `tools_tokens_estimate=0`, а `--mode run` показывает
ненулевую стоимость tools.

### 2. Сделать бюджетные действия отдельными событиями trace

Цель: сделать обрезку контекста видимой как последовательность решений, а не
только как итоговое состояние внутри `context_report`.

Архитектурный ориентир: trace должен быть журналом решений, а не только снимком
последнего состояния. В промышленных системах такие события нужны для аудита,
регрессий и объяснения поведения агента на длинных сессиях.

Задание: сейчас clipping tool output и удаление старой истории видны в
`context_report`, который лежит внутри `model_call_started`. Добавьте отдельные
trace-события, например `context_tool_output_clipped`, `context_history_dropped`
и `context_hard_limit_exceeded`, чтобы по JSONL было видно, какие действия были
применены перед model call.

События должны появляться одинаково для `ask` и `run`, если соответствующее
бюджетное действие произошло. Не дублируйте логику в двух местах: вынесите
запись событий в общий helper или сделайте сборку контекста возвращающей явный
результат бюджетных действий.

Ограничения:

- не дублировать большой tool output;
- не писать полный prompt;
- события должны появляться только когда действие реально произошло;
- сохранить текущий `context_report`.

Критерий готовности: trace показывает не только финальную сводку, но и
отдельные факты: что было обрезано, что было удалено и на каком ходу.

Проверка: смоделировать `run` с большим результатом `run_shell` и малым
бюджетом; убедиться, что в trace появились события clipping и, при
необходимости, удаления старой истории. Отдельно проверить `ask` со слишком
большим стартовым контекстом, чтобы `context_hard_limit_exceeded` и
`context_error` оставались согласованными.

## Продвинутый уровень

### 1. Суммаризировать удаляемую историю через модель

Цель: исследовать компромисс между размером контекста и сохранением рабочей
памяти агента при длинной сессии.

Архитектурный ориентир: это упрощённое сжатие рабочей памяти. Слой контекста
решает, какие элементы не помещаются, но не должен сам становиться клиентом
модели: LLM-вызов для summary должен жить на границе агентского цикла или в
явно переданном summarizer-интерфейсе.

Задание: сейчас старые `HistoryEntry` при нехватке бюджета удаляются целиком.
Добавьте механизм суммаризации истории перед удалением: когда контекст не
помещается, harness выбирает старые элементы истории, отправляет их модели
отдельным коротким запросом на summary, а затем заменяет исходные тяжёлые
сообщения компактным служебным фрагментом.

Не импортируйте `ModelClient` внутрь `ContextManager`. Один приемлемый вариант:
`ContextManager` возвращает кандидатов на сжатие, а `model_loop.py` вызывает
отдельный `HistorySummarizer` без tools и передаёт обратно готовый summary.
Другой вариант — явный callback, который можно подменить fake model client в
тесте. В обоих случаях граница остаётся прозрачной: слой контекста хранит и
рендерит данные, а обращение к модели управляется агентским циклом.

В следующих model calls агент должен видеть не полный старый tool output, а
краткое резюме: какие действия уже были выполнены, какие файлы или инструменты
затрагивались, какие выводы важны для продолжения работы.

Ограничения:

- summary-запрос должен быть коротким и не использовать tools;
- `ContextManager` не должен зависеть от `ModelClient` или HTTP-слоя;
- полная исходная история должна оставаться в JSONL trace;
- summary должно быть явно помечено как сжатая история, а не как новое
  пользовательское сообщение;
- если summary тоже не помещается, жёсткий бюджет важнее;
- trace должен показывать, что история была суммаризирована, какие entries
  заменены и сколько символов удалось сэкономить.

Критерий готовности: при длинной сессии старые tool turns заменяются
model-generated summary, и последующие вызовы модели продолжают работу,
опираясь на это резюме вместо полного старого контекста.

Проверка: создать fake model client для summary-запроса, несколько старых tool
turns и маленький бюджет. Проверить, что следующий `messages` содержит summary,
не содержит длинный stdout, а trace фиксирует событие суммаризации.
Исследовательская часть: сравнить тот же сценарий без summary и с summary;
кратко объяснить, сколько токенов удалось сэкономить и какая информация
потерялась.

### 2. Спроектировать priority-aware eviction для истории

Цель: перейти от простой стратегии «удаляем самые старые entries» к объяснимой
политике удержания важных наблюдений.

Архитектурный ориентир: это policy-based eviction. История удаляется не просто
по возрасту, а по явной политике, где важность, свежесть и жёсткий бюджет
разведены как разные факторы.

Задание: сейчас `ContextFragment.priority` влияет на порядок фрагментов, но
история удаляется в основном по возрасту. Добавьте приоритеты для элементов
истории или tool observations, чтобы важные результаты можно было сохранять
дольше обычных.

Один из простых вариантов: у `HistoryEntry` появляется `priority` со значением
по умолчанию, а eviction сначала удаляет старые entries с низкой важностью,
сохраняя недавние и важные элементы, пока это не нарушает жёсткий
`context_max_tokens`. Явно опишите шкалу: например, меньшее число означает более
важную историю, обычная история получает `100`, а равные приоритеты удаляются по
возрасту. Источник приоритета тоже должен быть явным: параметр
`record_assistant()`/`record_tool_result()` или внутреннее metadata-поле
observation, которое не утекает в сообщение модели.

Ограничения:

- приоритет не должен позволять превысить жёсткий бюджет;
- assistant/tool пара должна оставаться атомарной;
- metadata приоритета не должна попадать в `role=tool` content для модели;
- поведение должно быть видно в `context_report`;
- старые тесты бюджета должны остаться осмысленными.

Критерий готовности: на маленьком сценарии видно, что низкоприоритетная история
удаляется раньше, а важная сохраняется до тех пор, пока это совместимо с
бюджетом. В `context_report` видно, какой priority был у включённых и удалённых
entries и почему правило разрешения равенства выбрало именно их.

Проверка: создать три history entries: старую важную, старую обычную и новую
обычную. При малом бюджете должна удалиться старая обычная, а report должен
объяснить решение.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

После выполнения задачи проверьте:

- какие фрагменты попали в запрос модели;
- какие observations попали в историю;
- что было обрезано, суммаризировано или удалено;
- почему итоговый запрос поместился или не поместился в бюджет.
