# Лабораторные работы: 05-mcp

Эта ветка тренирует подключение внешних инструментов через MCP. Главный учебный
фокус: внешний процесс должен выглядеть для модели как обычный инструмент
харнесса, но сам харнесс обязан явно управлять запуском, именованием, обработкой
ошибок, окружением, ответами протокола и завершением процесса.

## Базовый уровень

### 1. Добавить проверку `mcp.json` без запуска серверов

Цель: отделить проверку настроек MCP от запуска внешних процессов и увидеть,
какие ошибки можно выявить заранее.

Задание: добавьте команду вроде `madharness-mini mcp validate`, которая читает
`.madharness-mini/mcp.json`, проверяет структуру файла и показывает список
серверов без запуска процессов. В выводе должны быть имя сервера, включён он
или нет, команда, рабочий каталог, таймаут и имена переменных окружения без
значений.

Если файл отсутствует, команда должна прямо сообщить, что MCP выключен для
проекта. Если сервер выключен через `enabled: false`, его нельзя запускать и
нельзя проваливать всю проверку из-за отсутствующей команды или невалидных
аргументов внутри этого выключенного блока.

Ограничения:

- команда не должна запускать `subprocess.Popen`;
- значения `env` не должны печататься в терминал или трассу;
- включённые серверы должны проходить существующую проверку `cwd`;
- основной `.madharness-mini/config.json` менять не нужно.

Критерий готовности: по выводу команды понятно, включён ли MCP, какие серверы
будут запускаться и какие настройки ошибочны, при этом ни один внешний процесс
не запускается.

Проверка: создать корректный `mcp.json`, файл с ошибкой JSON, сервер с
`enabled: false`, сервер с небезопасным `cwd` и сервер с нестроковым `env`.
Проверить код возврата, текст ошибки и отсутствие созданного дочернего процесса.

### 2. Показать карту зарегистрированных MCP-инструментов

Цель: сделать видимым переход от ответа `tools/list` к именам инструментов,
которые получает модель.

Задание: после успешного `tools/list` запишите в трассу отдельное событие,
например `mcp_tools_registered`. В нём должны быть имя MCP-сервера, исходные
имена инструментов, экспортированные имена вида `mcp__server__tool`, общее
количество инструментов и признак обрезки списка, если инструментов слишком
много.

Команда `trace` должна показывать короткую сводку по MCP: какие серверы
запустились и какие экспортированные имена добавлены в набор инструментов.
Полные схемы параметров в сводку выводить не нужно.

Где смотреть в спецификации:

- [Tools](https://modelcontextprotocol.io/specification/2025-11-25/server/tools),
  разделы `Listing Tools`, `Tool` и `Tool Names`.

Ограничения:

- не записывать большие полные схемы `inputSchema` в сводку;
- не менять формат имени `mcp__server__tool`;
- если список обрезан, это должно быть явно видно;
- ошибки при обработке отдельного инструмента должны указывать имя сервера.

Критерий готовности: по трассе можно понять, откуда появился внешний инструмент
и под каким именем модель могла его вызвать.

Проверка: использовать тестовый MCP-сервер с несколькими инструментами, включая
имя с точкой или пробелом. Проверить событие в трассе, вывод
`madharness-mini trace` и стабильность экспортированных имён.

## Основной уровень

### 3. Поддержать постраничный `tools/list`

Цель: увидеть, что список инструментов MCP-сервера по спецификации может
приходить несколькими порциями, и научить харнесс собирать все страницы
результата.

Задание: сейчас `StdioMcpClient.start()` вызывает `tools/list` один раз, поэтому
клиент обрабатывает только первую страницу ответа и регистрирует только
инструменты из неё. Добавьте поддержку `nextCursor`: если сервер вернул курсор
следующей страницы, клиент должен повторить `tools/list` с этим курсором и
объединить инструменты всех страниц.

В трассе должно быть видно, сколько страниц `tools/list` прочитано у каждого
сервера, сколько инструментов получено и был ли применён защитный лимит. Если
сервер повторно возвращает один и тот же курсор или выдаёт слишком много
страниц, харнесс должен остановиться с понятной ошибкой.

Где смотреть в спецификации:

- [Pagination](https://modelcontextprotocol.io/specification/2025-11-25/server/utilities/pagination);
- [Tools](https://modelcontextprotocol.io/specification/2025-11-25/server/tools),
  раздел `Listing Tools`.

Ограничения:

- одна страница без `nextCursor` должна работать как раньше;
- порядок инструментов между страницами должен сохраняться;
- нужен лимит на число страниц или общее число инструментов;
- при ошибке чтения очередной страницы процесс сервера должен закрываться.

Критерий готовности: MCP-сервер с большим числом инструментов регистрируется
полностью, а зацикленная или слишком длинная постраничная выдача не заставляет
запуск `run` зависнуть.

Проверка: подготовить тестовый MCP-сервер, который возвращает `tools/list` в две
страницы, и сервер, который повторяет один и тот же `nextCursor`. Проверить
итоговый список схем инструментов, порядок имён, события трассы и закрытие
процесса при ошибке.

### 4. Сделать ответы MCP-инструментов полезнее для следующего шага

Цель: улучшить преобразование `tools/call` в обычный ответ инструмента
харнесса, не передавая модели лишние данные.

Задание: улучшите `mcp_result_to_observation`. Ответ должен кратко показывать,
какой сервер и какой исходный инструмент был вызван, был ли результат ошибочным,
сколько текстовых блоков получено, был ли текст обрезан, есть ли
`structuredContent`, изображения, аудио или ссылки на ресурсы.

Для `isError: true` итоговый `summary` должен содержать короткую причину, если
её можно взять из текстового блока. Для изображений, аудио и ресурсов
по-прежнему передаются только метаданные, без строки `base64` и без попытки
скачать ресурс.

Где смотреть в спецификации:

- [Tools](https://modelcontextprotocol.io/specification/2025-11-25/server/tools),
  разделы `Calling Tools` и `Tool Result`.

Ограничения:

- большой текст должен проходить через общий механизм обрезки;
- строка `base64` из `image` и `audio` не должна попадать в ответ модели;
- в этой задаче не добавлять отдельные методы `resources/list` или
  `resources/read`;
- ответ должен оставаться обычным JSON-совместимым словарём.

Критерий готовности: модель получает достаточно информации, чтобы решить, нужно
ли запрашивать данные дальше, повторить вызов с другими аргументами или
остановиться из-за ошибки MCP-сервера.

Проверка: расширить тестовый MCP-сервер ответами с несколькими текстовыми
блоками, `structuredContent`, `isError: true`, изображением и ресурсом.
Проверить `ok`, `summary`, признаки обрезки и отсутствие строки `base64` в ответе
инструмента и трассе.

## Продвинутый уровень

### 5. Добавить минимальную поддержку ресурсов и промптов MCP

Цель: выйти за пределы одних только инструментов и реализовать следующий слой
серверных возможностей MCP: `resources` и `prompts`.

Задание: сейчас харнесс использует MCP-сервер только как источник инструментов:
`tools/list` и `tools/call`. Добавьте минимальную поддержку серверных
возможностей `resources` и `prompts` из спецификации MCP. После `initialize`
клиент должен проверять объявленные возможности сервера и, если они есть, уметь
выполнить `resources/list`, `resources/read`, `resources/templates/list`,
`prompts/list` и `prompts/get`.

Сами ресурсы, шаблоны ресурсов и промпты не нужно автоматически добавлять в
первое обращение к модели. Дайте модели отдельные локальные инструменты-обёртки:
посмотреть ресурсы сервера, прочитать ресурс по `uri`, посмотреть шаблоны
ресурсов, посмотреть промпты и получить промпт по имени с аргументами.
Результаты должны превращаться в обычные ответы инструментов харнесса с
обрезкой больших текстов и без передачи бинарных данных в виде `base64`.

В документации явно укажите, что это не полная реализация спецификации MCP:
клиентские возможности `roots`, `sampling` и `elicitation` остаются
неподдержанными и должны отвечать понятной ошибкой, если сервер их запрашивает.

Где смотреть в спецификации:

- [Server Features](https://modelcontextprotocol.io/specification/2025-11-25/server/index);
- [Resources](https://modelcontextprotocol.io/specification/2025-11-25/server/resources);
- [Prompts](https://modelcontextprotocol.io/specification/2025-11-25/server/prompts);
- [Pagination](https://modelcontextprotocol.io/specification/2025-11-25/server/utilities/pagination);
- [Lifecycle](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle).

Ограничения:

- не загружать все ресурсы автоматически при запуске сервера;
- `resources/list`, `resources/templates/list` и `prompts/list` должны
  поддерживать `nextCursor`;
- бинарное содержимое ресурса передавать только метаданными;
- нужна защита от слишком большого ресурса или слишком длинного промпта;
- не объявлять поддержку `roots`, `sampling` или `elicitation`, если она не
  реализована.

Критерий готовности: модель может по запросу увидеть, какие ресурсы, шаблоны
ресурсов и промпты предоставляет MCP-сервер, прочитать выбранный ресурс или
получить выбранный промпт, но контекст не раздувается всем содержимым сервера
заранее.

Проверка: расширить тестовый MCP-сервер: добавить `resources/list`,
`resources/read`, `resources/templates/list`, `prompts/list`, `prompts/get`,
постраничную выдачу и бинарный ресурс. Проверить новые локальные инструменты,
обрезку текста, отсутствие строки `base64` в ответе модели, события трассы и
понятную ошибку для неподдержанных клиентских запросов.

### 6. Сделать жизненный цикл MCP-процесса устойчивым к зависаниям

Цель: проверить, что внешний процесс не остаётся работать после ошибки модели,
исчерпания ходов или зависшего MCP-запроса.

Задание: усильте управление жизненным циклом MCP-сервера. В трассе должны быть
видны запуск, таймаут запроса, попытка штатного закрытия, принудительное
завершение и итоговый код выхода. Отдельно проверьте, что
`ToolRegistry.close()` вызывается при ошибке обращения к модели, при превышении
`max_turns` и при таймауте `tools/call`.

Для зависшего сервера подготовьте тестовый процесс, который работает через
стандартные потоки (`stdio`) и отвечает на `initialize`, но зависает на
`tools/list` или `tools/call`. Харнесс должен вернуть понятную ошибку, закрыть
процесс и не потерять исходную причину сбоя.

Где смотреть в спецификации:

- [Lifecycle](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle),
  разделы про инициализацию, рабочую фазу и завершение соединения.

Ограничения:

- тестовый сервер должен быть на стандартной библиотеке Python;
- тесты не должны ходить в сеть и не должны требовать `npx` или `uvx`;
- закрытие должно быть идемпотентным;
- трасса не должна раздуваться полным `stdout` или `stderr` сервера.

Критерий готовности: после любого сценария завершения не остаётся дочерних
MCP-процессов, а по трассе понятно, почему процесс был закрыт штатно или
принудительно.

Проверка: добавить тестовые MCP-серверы для обычного завершения, ошибки модели,
превышения `max_turns`, зависшего `tools/list` и зависшего `tools/call`.
Проверить маркеры закрытия, код выхода и события трассы.

### 7. Добавить управляемые долгоживущие процессы для инструментов

Цель: дать агенту возможность запускать локальный сервис или бэкенд не как
одноразовую shell-команду, а как управляемый процесс, который живёт между ходами
модели и закрывается при завершении работы харнесса.

Задание: сейчас `run_shell` использует `subprocess.run(..., timeout=60)` и
подходит только для команд, которые завершаются сами. Поэтому агент не может,
например, запустить локальный HTTP-сервер, проверить его через MCP-инструмент
или другой инструмент, исправить код и повторить проверку в том же `run`.

Добавьте минимальный менеджер долгоживущих процессов и отдельные инструменты
для работы с ним: запустить процесс, посмотреть статус, прочитать новые данные
из `stdout`/`stderr`, дождаться завершения с коротким таймаутом и остановить
процесс.
Запуск должен возвращать стабильный `process_id` или имя процесса, которое
модель сможет использовать в следующих вызовах инструментов. Реестр процессов
должен существовать только внутри одного запуска `run` и очищаться через общий
жизненный цикл `ToolRegistry.close()`.

В трассе должны быть видны запуск процесса, рабочий каталог, команда без
раскрытия секретов, чтение вывода, штатная остановка, принудительное завершение
и итоговый код выхода. Большой `stdout`/`stderr` нужно обрезать тем же способом,
что и обычные ответы инструментов, а при повторном чтении не возвращать уже
переданный вывод.

Ограничения:

- не превращать `run_shell` в произвольный интерактивный терминал;
- команды долгоживущих процессов должны проходить существующую shell-политику и
  проверку `cwd`;
- процесс нельзя запускать вне `workspace_root`;
- нужен лимит на число одновременно живущих процессов;
- закрытие должно быть идемпотентным и срабатывать при успехе, ошибке модели,
  превышении `max_turns` и исключениях во время вызова инструмента;
- нельзя записывать в трассу полный поток вывода или значения секретных
  переменных окружения.

Критерий готовности: агент может в одном `run` поднять небольшой локальный
сервер, дождаться признака готовности в `stdout` или через проверочный запрос,
выполнить несколько последующих обращений к этому серверу, а затем штатно
остановить процесс. После завершения харнесса не остаётся дочерних
процессов.

Проверка: подготовить тестовый сервер на стандартной библиотеке Python, который
пишет строку готовности, отвечает на локальный запрос и корректно реагирует на
завершение. Проверить запуск, чтение инкрементального вывода, повторную
проверку после изменения кода, ручную остановку, автоматическую очистку в
`ToolRegistry.close()`, лимит процессов и сценарий зависшего процесса, который
приходится завершать принудительно.

## Проверка

Запускайте тесты из корня репозитория:

```bash
uv run -m unittest discover -s tests
```

После выполнения задачи проверьте:

- какие MCP-серверы были запущены;
- какие имена инструментов получила модель;
- что попало в трассу;
- какие секреты не ушли во внешний MCP-процесс;
- закрылись ли дочерние процессы после успешных и ошибочных запусков;
- закрылись ли долгоживущие процессы инструментов после успешных и ошибочных
  запусков.
